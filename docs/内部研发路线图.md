# IClude 内部研发路线图

## 1. 研发前提

当前共识先锁定：

- `Phase 2` 已完成
- 当前产品已可提供给日常用户
- 后续研发不再用“Phase 3 / Phase 4”混合承载所有目标
- 后续工作拆成 `Benchmark Track` 和 `Enterprise Track`
- `Hindsight 91.4%` 是长期技术标杆，不是发布日期承诺

核心判断：

- 对开源主线最重要的，不是再加多少企业功能
- 而是先把 **检索质量 + 记忆利用率 + LongMemEval 可复现评测** 做扎实

---

## 2. 总目标

### 2.1 Benchmark Track

目标：

- 建立可复现 LongMemEval 评测体系
- 把 IClude 从“检索骨架不错”推进到“端到端记忆质量稳定”
- 先做到 `80%+`
- 再争取 `85%+`
- 长期逼近 `Hindsight 91.4%`

### 2.2 Enterprise Track

目标：

- 企业交付不阻塞开源主线
- 所有企业能力按商业价值单独排期

---

## 3. 路线原则

### 3.1 研发排序原则

按以下顺序决策优先级：

1. 是否直接提升 LongMemEval 端到端表现
2. 是否直接改善日常用户真实体验
3. 是否属于必要基础设施
4. 是否只是企业补充能力

### 3.2 不再混做的内容

以下两类工作不再放在同一个迭代里绑定交付：

- Benchmark 优化
- 企业能力建设

原因：

- 前者追求记忆质量
- 后者追求治理、运维和交付
- 两者都重要，但节奏完全不同

---

## 4. Benchmark Track 研发清单

### B1. 评测闭环

目标：先把尺子立起来。

任务：

- 建立 `testing/eval/`
- 接入 LongMemEval 官方数据集
- 输出 `accuracy / recall@k / MRR / NDCG / category breakdown`
- 固化 3 条基线：
  - full-context
  - 当前 hybrid retrieval
  - reflect end-to-end
- 支持 CI 或至少本地脚本稳定复跑

验收标准：

- 每次优化前后都能出同口径对比结果
- category breakdown 能定位弱项，不只看总分

### B2. 精排能力

目标：解决“召回到了，但排序不够准”。

任务：

- 新增 `Reranker` 接口
- 引入 cross-encoder 重排
- 管道固定为：
  - 三路检索
  - RRF
  - Cross-encoder
  - 强度加权
  - Token 裁剪
- 支持本地 ONNX 或远程 rerank API

验收标准：

- 在固定 benchmark 上优于当前 hybrid baseline
- 不允许只提升个别 query 而拉低整体稳定性

### B3. Reflect 利用率

目标：解决“检索到了但没用好”。

任务：

- Reflect 多轮累积前几轮 reasoning 和证据摘要
- 每轮 Top-K 不再固定写死，改为按 intent / token budget / round 调整
- 在关键路径引入精确 token 计算

验收标准：

- 多轮问题 category 有实测提升
- token 裁剪不再频繁误伤有效证据

### B4. 时间与演化

目标：补齐 LongMemEval 常见失分点。

任务：

- Temporal query 动态时间窗口
- Consolidation 候选从静态列表改为时间轮转 + 随机采样
- 补齐三层演化：
  - fact
  - observation
  - mental_model
- 新增 `layer` 和 `derived_from`

验收标准：

- temporal 类问题显著改善
- observation / mental_model 真正进入 recall 主链路，而不只是存着不用

### B5. 结构化约束

目标：让不同场景下的记忆系统更可控。

任务：

- `mission / directives / disposition` 接入 Reflect
- Entity/Relation 类型配置化

验收标准：

- 不同 context 下表现更稳定
- 配置可控，不靠硬编码调 prompt

## 5. Benchmark Track 里程碑

### W1-W2

- LongMemEval 评测框架可跑
- 产出第一版基线
- 明确 category 弱项排序

### W3-W4

- Cross-encoder 上线
- Temporal 动态窗口上线
- 第一次系统性超过当前 hybrid baseline

### W5-W6

- Reflect 历史累积上线
- 精确 token 计算上线
- 多轮与复杂问题 category 进入稳定迭代

### W7-W8

- 三层记忆演化接入 recall 主链路
- 判断是否具备冲击 `85%+` 的条件

## 6. Enterprise Track 研发清单

这条路线独立排期，不阻塞 Benchmark Track。

### E1. 接入与治理

- 认证与权限
- Go SDK
- Node.js SDK
- Prometheus 监控

### E2. 可视化与运维

- 管理后台
- 性能压测与稳定性优化
- 架构加固
- 标签体系升级

### E3. 企业版长期能力

- 多租户物理隔离
- 分布式后端（PostgreSQL 替换 SQLite）
- 安全合规
- SSO / SLA / 私有化交付

### E4. 通用记忆层协议（Universal Memory Layer）

> 详细设计见 [Universal Memory Layer Design Spec](../superpowers/specs/2026-04-02-universal-memory-layer-design.md)

基于 B4 已实现的 `memory_class` / `derived_from` 字段，增量实施完整的 4 层记忆协议：

- Core Memory Blocks：小容量强约束工作记忆，始终注入上下文
- 受控写入策略：外部工具默认只写 `episodic`，`core` / `procedural` 需晋升
- 晋升管道：episodic → candidate → semantic / procedural / core
- Scope 优先级检索：session > project > user，跨工具共享 `user/*` core
- API/MCP 扩展：`capture_mode`、`candidate_for`、`include_core`、`iclude_read_core`、`iclude_promote_memory`

前置依赖：B4 完成（memory_class + derived_from 字段）、E3 PostgreSQL 迁移

---

## 7. 研发决策规则

如果一个需求同时满足以下任一条件，优先进入 Benchmark Track：

- 能直接提升 LongMemEval
- 能直接改善日常用户“记忆答复质量”
- 能降低 benchmark 回归成本

如果一个需求主要提升以下价值，则归入 Enterprise Track：

- 权限治理
- 运维可见性
- 商业交付
- 企业隔离与合规

---

## 8. 当前研发口径

内部统一口径建议如下：

> IClude 当前已经完成日常用户可交付版本。后续研发以 Benchmark Track 为主线，对标 Hindsight 91.4%，优先补齐评测、精排、Reflect 利用率和时间演化能力；Enterprise Track 独立推进，不阻塞主线。

---

## 9. 成功标准

### 短期

- LongMemEval 可复现评测流程跑通
- 能稳定追踪每次优化增益
- 端到端达到 `80%+`

### 中期

- 端到端达到 `85%+`
- 复杂、多轮、时间类问题有明确提升

### 长期

- 逼近 `Hindsight 91.4%`
- 在不牺牲本地优先和日常用户体验的前提下形成差异化
