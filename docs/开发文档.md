## 1. 项目定位与核心原则

### 1.1 一句话定义
IClude 是一个开源、本地优先、混合存储的企业级记忆体系统，专注于为 AI 应用提供记忆的存储、检索与思考型检索能力，不涉及生成、外部工具或用户界面。

### 1.2 核心范围（做什么）
- ✅ 记忆存储：支持结构化元数据 + 向量嵌入的混合存储（SQLite + Qdrant）
- ✅ 记忆检索：向量检索、全文检索、图谱关联检索、三路 RRF 混合融合
- ✅ 记忆反思（Reflect）：调用外部 LLM 对已有记忆做多步推理，自动生成新洞察（kind=mental_model）
- ✅ 自动知识抽取（Retain）：写入记忆时可选由 LLM 自动提取实体/关系/时间，写入知识图谱
- ✅ 记忆演化：事实(fact) → 经验(observation) → 心智模型(mental_model) 三层自动演化
- ✅ 记忆生命周期：5 级保留策略 + 指数衰减 + 强化 + 软删除/恢复 + 过期清理
- ✅ 层级上下文：树形 Context 结构（物化路径），支持行为配置（mission/directives/disposition）
- ✅ 知识图谱：实体 + 关系 + 记忆-实体关联，支持图谱遍历检索
- ✅ 文档处理：上传 → 分块 → 记忆化管道
- ✅ 分区治理：多团队数据隔离，支持跨团队统一检索
- ✅ 开放 API + SDK + MCP：RESTful API、Python/Go/Node.js SDK、MCP Server（Claude/Gemini 直接调用）

### 1.3 明确不做的范围（不做什么）
- ❌ 不生成答案：调用外部 LLM 仅用于知识抽取和记忆反思，不生成面向用户的自然语言回答
- ❌ 不执行外部工具：不集成代码执行、网络搜索、API 调用等
- ❌ 不内置 AI 模型：用户自行配置 API Key 或本地模型（支持 OpenAI/Anthropic/Ollama 等）

### 1.4 设计哲学
- 本地优先：数据默认存储在用户自己的机器或服务器，一个二进制文件搞定
- 透明可审计：数据格式开放，支持人工检查和迁移
- 渐进式复杂：从单机单文件开始，按需演进至分布式高可用
- 一人可维护：所有设计考虑单人开发与运维成本

## 3. 核心模块说明

### 3.1 存储抽象层（Storage Abstraction Layer）
职责：定义统一的存储接口，隔离底层实现。

#### 接口定义（Go 示例）：

```go
type MemoryStore interface {
    Create(mem *Memory) error
    Get(id string) (*Memory, error)
    Update(mem *Memory) error
    Delete(id string) error
    SearchText(query string, limit int) ([]*Memory, error) // 全文检索
}

type VectorStore interface {
    Insert(memoryID string, embedding []float32, payload map[string]interface{}) error
    Delete(memoryID string) error
    Search(embedding []float32, limit int) ([]SearchResult, error)
}
```

#### 实现：
- SQLiteStore：基于 SQLite，负责元数据、全文检索（FTS5）、实体关系表。
- QdrantStore：基于 Qdrant REST API，负责向量存储与检索。

#### 配置开关：通过环境变量或配置文件控制是否启用每个后端。

### 3.2 业务逻辑层

#### 3.2.1 Memory Manager（✅ Phase 1 已完成）
- 记忆 CRUD，根据配置同时写入 SQLite 和/或 Qdrant（最佳努力双写）
- 创建时支持可选字段：context_id, kind, scope, tags, retention_tier 等
- 标签关联：find-or-create + 记忆-标签多对多
- 软删除/恢复 + Context memory_count 维护
- 对话摄取（IngestConversation）：批量写入 + 自动 turn_number

#### 3.2.2 Retriever（✅ Phase 1 已完成，Phase 2~3 增强）

**Phase 1 — 双路 RRF：**
- 三模式：SQLite-only / Qdrant-only / Hybrid RRF(k=60)
- 强度加权：结果分数 × effective_strength（指数衰减）
- 过期过滤 + SearchFilters 多条件过滤

**Phase 2 — 三路 RRF + Token 裁剪 [B3][B5]：**
- 新增第三路：Graph 实体关联检索（查询→实体识别→图谱遍历→关联记忆）
- Token 感知裁剪（max_tokens 参数，按预算截断）

**Phase 3 — Cross-encoder 重排 [B4]：**
- RRF 融合后可选调用 Reranker 精排
- 完整管道：三路检索 → RRF → Reranker → 强度加权 → Token 裁剪

#### 3.2.3 Context Manager（✅ Phase 1 已完成）
- 树形上下文管理（物化路径），支持 project / topic / session 三种类型
- CRUD + ListChildren + ListSubtree + Move（含循环引用检测）
- Phase 3 扩展：行为配置（mission/directives/disposition）[B7]

#### 3.2.4 Graph Manager（✅ Phase 1 已完成，Phase 2 增强）
- Entity/Relation/MemoryEntity 全套 CRUD
- Phase 2 增强：Extractor 自动抽取（Create 时 auto_extract=true）[B2]

#### 3.2.5 Reflect Engine（Phase 2 新增）[B1]

借鉴 Hindsight 的 Reflect 能力，与原规划"多轮思考型检索"合并设计。

- 输入：用户问题、scope、最大轮数、token 预算
- 过程：
  1. 召回相关记忆（调用 Retriever）
  2. 调用外部 LLM 分析记忆间关联，判断是否需要更多信息
  3. 若需要，生成新查询关键词并再次检索
  4. 循环直到满足条件或达到最大轮数
  5. 最终推理结果自动写回为新记忆（kind=mental_model, source_type=reflect）
- 输出：

```json
{
  "result": "多步推理得出的综合分析",
```

#### 3.2.6 Extractor（Phase 2 新增）[B2]

自动知识抽取器，写入记忆时由 LLM 提取结构化知识。

- 集成点：`manager.Create(mem, auto_extract=true)` 时自动调用

#### 3.2.7 Lifecycle Manager（✅ Phase 1 已完成）
- 5 级保留策略：permanent / long_term / standard / short_term / ephemeral
- 指数衰减：`effective = strength × e^(-decay_rate × hours)`
- 强化公式：`strength += 0.1 × (1 - strength)`
- 过期清理 + 硬删除级联（memory_tags + memory_entities）

### 后续路线调整（2026-04-01）

结论先定下来：**Phase 2 已经是面向日常用户的可交付版本。**

后续工作不再把“对标 Hindsight 91.4%”和“企业版建设”塞进同一条 Phase，而是拆成两条独立路线：

1. **Benchmark Track（开源主线）**
   目标：围绕 LongMemEval 持续优化 recall + reasoning，让日常用户也能直接受益于更强的记忆质量。
2. **Enterprise Track（商业主线）**
   目标：补齐认证、后台、监控、多租户和企业交付能力，但不阻塞 Benchmark Track。

这样拆分的原因：
- `Hindsight 91.4%` 是技术标杆，不是企业功能清单。
- 日常用户关心的是“记得住、找得到、答得对”，不是 SSO、审计或 Helm Chart。
- 企业项混进主路线，会稀释 LongMemEval 对标工作，导致路线看起来很大，但对分数帮助很小。

## 8. 配置与部署

### 8.1 配置文件示例（YAML）

```yaml
storage:
  sqlite:
    enabled: true
    path: "./data/iclude.db"
  qdrant:
    enabled: true
    url: "http://localhost:6333"
    collection: "memories"
    dimension: 384

partitions:
  enabled: true
  catalog_path: "./data/partitions.db"  # 分区目录库

llm:  # 多轮检索所需
  default_provider: "openai"
  openai:
    api_key: "${OPENAI_API_KEY}"
    model: "gpt-4"

server:
  port: 8080
  auth_enabled: false  # Phase 3 启用
```

### 8.2 部署方式
- 单二进制文件：直接运行，SQLite 文件生成在指定目录。
- Docker 单容器：将核心服务和 Qdrant 用 docker-compose 一起启动。
- Kubernetes：提供 Helm Chart（Phase 3 后）。

### 风险控制：
- 技术风险：每个阶段都有可运行版本，不跳过测试。LLM 调用失败不影响核心存取。
- 性能风险：Reflect/Extractor 依赖外部 LLM，设置超时和重试。图谱检索可配置开关。
- 成本风险：LLM 调用按需付费，auto_extract 和 reflect 均为可选功能，不强制。
- 市场风险：对齐 Hindsight 的 retain/recall/reflect 命名，降低用户迁移门槛。
