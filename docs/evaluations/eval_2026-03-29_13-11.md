## 数据库评估报告

### 总体评分：7.5 / 10

该项目数据库层整体设计扎实，迁移机制规范、FTS5 集成完整、事务边界清晰。主要扣分点集中在：FK 约束缺失、文档表的 TOCTOU 竞态、OFFSET 分页、连接池配置不完整，以及 V9→V10 迁移的安全性疏漏。

### 关键发现

#### CRITICAL

**C-1：全库无外键约束（REFERENCES / ON DELETE）**
所有关联表均无 `FOREIGN KEY ... REFERENCES` 声明，尽管 `PRAGMA foreign_keys=ON` 已设置。删除记录后，关联表中的孤儿行永久残留。

**C-2：`sqlite_document.go` 的 content_hash 预检查存在 TOCTOU 竞态**
`Create()` 方法在插入前先执行 `SELECT COUNT(*)`，两步之间没有事务保护。应删除该检查，仅依赖唯一约束的错误响应。

#### HIGH

- **H-1**：`memory_tags` 表缺少 `tag_id` 反向索引，`GetMemoriesByTag` 查询会全表扫描
- **H-2**：`async_tasks` 表缺少 `SKIP LOCKED` 模式支持，并发消费可能重复处理
- **H-3**：`List`/`ListByContext` 使用 OFFSET 分页，大偏移量性能差
- **H-4**：V9→V10 迁移缺少幂等保护（`isColumnExistsError`），与其他版本不一致

#### MEDIUM

- 连接池缺少 `synchronous=NORMAL`、`cache_size` 等性能 PRAGMAs
- `documents` 状态部分索引过于限制（仅覆盖 pending/processing）
- `ListByStatus` 缺 `(status, created_at)` 复合索引
- `entities` 表 `COLLATE NOCASE` 索引与 LIKE 查询不兼容
- `Reinforce` 方法未使用事务

#### LOW

- `schema_version` 插入语法不统一
- `DB() interface{}` 缺乏类型安全
- 死代码 `migrateAddPerformanceIndexes`
- 文档硬删除与记忆软删除策略不一致

### 亮点

- FTS5 事务一致性完美：主表与虚拟表永不出现不一致
- BM25 权重 CTE 避免重复计算，标准最优写法
- 部分索引精准设计（软删除、哈希去重、过期时间）
- WAL + busy_timeout + mmap 标准配置
- 多租户可见性 SQL 抽象 `visibilityCondition()` 安全可复用

## AI/检索评估报告

### 总体评分：7.8 / 10

三路检索管线架构优秀，但存在文档分块无法进入向量检索的关键问题。

### 关键发现

#### CRITICAL

**C1：文档分块绕过 `Manager.Create()`，chunks 无 embedding**
`processor.go` 直接调用 `memStore.Create()` 而非 `Manager.Create()`，文档分块不会生成向量嵌入，对语义搜索完全不可见。

**C2：Chunker 的 `estimateTokens` 使用字节长度，overlap 切分产生无效 UTF-8**
`len(s)/3` 对 CJK 严重低估，字节级切分会截断多字节字符。

#### HIGH

- Qdrant `SearchFiltered` 忽略时间过滤器
- Graph 通道存在 N+1 查询问题（每次检索 20+ 次 SQLite 查询）
- Extractor 关系去重为 O(N) 全量扫描
- Consolidation 候选选择未过滤无向量记忆

#### MEDIUM

- Qdrant 可见性过滤存在逻辑缺口
- Reflect 引擎丢弃轮次间对话历史
- OpenAI embedder 硬编码端点 URL
- Chunker overlap 按字节而非句子边界切分
- RRF 忽略分数幅度
- 预处理器对混合语言查询分类不足

### 亮点

- 三路加权 RRF 融合架构设计优秀
- 意图感知查询预处理（CJK 调优阈值）
- 3 级 fallback 解析（JSON → regex → LLM → raw）
- 记忆生命周期系统完备（衰减、强化、晶化、归纳）
- Markdown 感知三层分块设计合理

## 安全评估报告

### 总体评分：7.2 / 10

整体安全架构合理，多层防御到位，但文件上传验证和授权检查存在可利用漏洞。

### 关键发现

#### CRITICAL

无 CRITICAL 级漏洞。

#### HIGH

- **H1**：文件类型验证仅依赖扩展名，无 Magic Bytes 检测（`mimetype` 库已在依赖中但未使用）
- **H2**：图谱资源（Entity/Relation）写操作缺少归属授权检查
- **H3**：MCP 服务器默认无认证（`api_token` 默认为空），生产误配置风险
- **H4**：`/reprocess` 端点缺少 owner 归属验证

#### MEDIUM

- CORS 默认通配符 `*`
- `document.name` 无长度上限、文件名无净化
- 速率限制基于 `ClientIP()`，可被 IP 欺骗
- 全局 4MB MaxBodySize 覆盖了文档上传的 100MB 配置
- 缺少 HSTS/CSP 安全响应头
- Document List 接口 `limit` 无上界
- MCP SSE 无最大会话数限制

#### LOW

- 文件/数据库目录权限 0755（应为 0700）
- List 过滤参数解析但未使用
- 未匹配错误未做服务端日志

### 亮点

- 恒定时间 API Key 比对（`crypto/subtle.ConstantTimeCompare`）
- 全库参数化 SQL，零注入风险
- FTS5 查询净化（`sanitizeFTS5Query`）
- 系统身份防冒充（`__system__` 拦截）
- 多层分级速率限制（20/10/2 rps）
- 路径遍历基本防护（`filepath.Base` + UUID 隔离）
- SHA-256 + `io.TeeReader` 零额外内存去重

## 架构评估报告

### 总体评分：7.5 / 10

项目展现成熟的 Go 工程实践和清晰的分层思维，feature gating 和依赖注入设计优秀。

### 关键发现

#### CRITICAL

无 CRITICAL 级问题。

#### HIGH

- **H1**：`MemoryStore` 接口过于庞大（27+ 方法），违反接口隔离原则
- **H2**：`search` 包对 `memory` 包的反向依赖（仅为 `ApplyStrengthWeighting` 一个函数）
- **H3**：`NewManager` 构造函数 9 个参数，超出可读性阈值

#### MEDIUM

- `Processor` 同时承担 CRUD 代理和处理管线双重职责
- `Processor.processDocument` 直接用 `memStore`，跳过 Manager 层的向量写入
- `bootstrap.Init()` 267 行线性增长风险
- `ParseRouter.ParserUsed()` 硬编码解析器名称
- 配置管理全局单例，测试隔离困难
- `ProcessAsync` 使用裸 goroutine 而非持久化任务队列

#### LOW

- `document` 包多个文件重复 package doc 注释
- `estimateTokens` 固定 1:3 比例，CJK 低估
- `LocalFileStore.Delete` 接受任意路径，无穿越防护

### 亮点

- 依赖流向严格无环
- Feature gating via nil checks 优雅降级
- 文档处理管线三层架构（Parser + ParseRouter + Chunker）设计精良
- bootstrap 单入口注入，main.go 极简
- 接口定义集中 + 实现扁平化
- LLM Fallback Chain 高可用设计
- Scheduler/Heartbeat config-gated 按需注册
- Router 三级分层限流

### 最高优先级行动项（TOP 3）

1. **[CRITICAL] 文档分块必须经过 `Manager.Create()` 而非直接 `memStore.Create()`**
   — 当前文档 chunks 不会生成向量嵌入，对语义搜索完全不可见。这是整个文档入库功能的核心 bug。修复方案：`Processor` 注入 `memory.Manager` 替代 `store.MemoryStore`。
   > 影响: AI/检索 C1 + 架构 M2

2. **[CRITICAL] V10 迁移添加幂等保护 + 消除 content_hash TOCTOU 竞态**
   — V10 是唯一缺少 `isColumnExistsError` 保护的迁移版本，会导致重跑失败。`Create()` 中的预检查 SELECT 是冗余且危险的。
   > 影响: 数据库 C-2 + H-4

3. **[HIGH] 文件上传安全加固：Magic Bytes 验证 + /reprocess 授权检查 + MaxBodySize 修复**
   — 三个安全问题可一次修复：使用已有的 `mimetype` 库验证文件内容类型，`/reprocess` 端点补充 owner 校验，修复全局 4MB 限制覆盖上传路由的问题。
   > 影响: 安全 H1 + H4 + M4
