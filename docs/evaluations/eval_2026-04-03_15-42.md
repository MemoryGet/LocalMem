## 数据库评估报告

### 总体评分：8.1 / 10（上次 7.4，+0.7）

上次 C1（迁移编排器 version 变量未更新）、C2（List 无 limit 上界）、H2（memory_handler limit 500）、H4（SearchTextFiltered 可见性绕过）均已修复。

---

### 关键发现

#### CRITICAL

无。

#### HIGH

**H1（持续）— 无数据库级外键约束**

所有关联表（`memory_tags`、`memory_entities`、`entity_relations`）在 DDL 中无 `FOREIGN KEY` 声明。应用层级联不完整：`SoftDelete`/`Delete` 单条不清理关联表，仅 `PurgeDeleted` 批量清理时处理。

**H3（新发现）— SearchText / SearchTextFiltered 无 limit 上界**

Store 层的 `SearchText`/`SearchTextFiltered` 只有下界保护（`<=0 → 10`），无上界。Retriever 层有 `>100` 截断，但直接调用 Store 的路径无保护。

#### MEDIUM

- M1: `ListMissingAbstract` 无上界 + `abstract` 列无部分索引
- M2: `ListEntities`/`FindEntitiesByName` Store 层无上界
- M3: OFFSET 分页在深翻页时性能退化
- M4: `document.List` Store 层无 limit 上界
- M5: `content_hash` 唯一索引的 partial 条件与 Restore 逻辑不一致

#### LOW

- L1: 无 `PRAGMA wal_autocheckpoint` 配置
- L2: 无定期 `PRAGMA optimize`
- L3: V9→V10 跳过 documents 表错误时仅 Debug 日志
- L4: `ListWeak`/`ListExpired` 无上界

### 与上次评估对比

| 问题 | 上次 | 当前 |
|------|------|------|
| C1: V9→V12 version 未更新 | CRITICAL | **已修复** |

| H2: limit 500 vs 200 | HIGH | **已修复** |
| H4: 可见性绕过 | HIGH | **已修复** |

### 改进建议

1. [P1] SearchText/SearchTextFiltered 加 limit 上界 200
2. [P2] SoftDelete/Delete 单条补全关联表清理
3. [P2] document.List 加 limit 上界
4. [P2] `abstract` 列加部分索引
5. [P3] 配置 WAL checkpoint 阈值
6. [P3] 定期 `PRAGMA optimize`

### 亮点

## AI/检索评估报告

### 总体评分：8.6 / 10（上次 8.2，+0.4）

上次 C1（goroutine panic recovery）和 C2（embed LimitReader）已完全修复。Consolidator 已改善（MemoryCreator 接口），但 fallback 路径仍保留。

---

### 关键发现

#### CRITICAL

无。

#### HIGH

**H1（降级）— Extractor 关系去重 TOCTOU**

`resolveRelation` 先查后写不在事务内，并发可创建重复关系。GraphStore 无 UNIQUE 约束兜底。影响降级：实体抽取是 best-effort 异步操作。

**H2（改善中）— Consolidator fallback 仍可直写 Store**

新增 `MemoryCreator` 接口 + `SetCreator()` 延迟注入，正常路径走 Manager。但 `creator == nil` fallback 仍保留。

**H3（新发现）— Heartbeat 晋升用 List(nil, 0, 200) 全量扫描**

仅返回最新 200 条，高频强化的旧记忆可能被遗漏。需专用查询按 `reinforced_count DESC` 排序。

#### MEDIUM

- M1: CachedEmbedder LRU touchLocked O(n)
- M2: agglomerativeClustering O(n³)
- M3: Reflect priorRounds 无总 token 估算
- M4: HyDE 用 context.Background() 绕过请求 ctx
- M5: Temporal 通道权重 1.2 硬编码不可配置

#### LOW

- L1: Embedding 维度未在应用层校验
- L2: adaptiveTopK 未使用 priorRounds 参数
- L3: selectCandidates rand.Shuffle 无确定性种子
- L4: Graph 遍历无 entity 数量上限

### 与上次评估对比

| 问题 | 上次 | 当前 |
|------|------|------|
| C1: asyncExtract 无 panic recovery | CRITICAL | **已修复** |
| C2: embed 客户端无 LimitReader | CRITICAL | **已修复（全覆盖）** |
| H3: Extractor TOCTOU | HIGH | 持续存在（降级） |
| H5: Consolidator 绕过 Manager | HIGH | **改善中** |

### 改进建议

1. [HIGH] entity_relations 加 UNIQUE 约束
2. [HIGH] Consolidator 删除 direct-store fallback
3. [HIGH] Heartbeat 晋升增加专用查询
4. [MEDIUM] HyDE context.Background() → ctx
5. [MEDIUM] Reflect priorRounds token 预算
6. [MEDIUM] CachedEmbedder 改用 container/list

### 亮点

- LimitReader 全覆盖（6 个外部 HTTP 客户端无遗漏）
- 五通道 RRF 融合（FTS + HyDE + Qdrant + Graph + Temporal）
- 查询预处理六步管道完整
- Reflect 引擎设计成熟（adaptive Top-K + prior rounds + query dedup + 4-level fallback）
- 记忆生命周期三重机制（衰减 + 访问 boost + 晋升）+ weightCap 2.0

## 综合总结

### 维度评分汇总

| 维度 | 上次 | 本次 | 变化 | 关键改进 |
|------|------|------|------|---------|
| 数据库 | 7.4 | **8.1** | +0.7 | C1/C2/H2/H4 修复，迁移拆分 |
| AI/检索 | 8.2 | **8.6** | +0.4 | panic recovery + LimitReader 全覆盖 |
| 安全 | 8.1 | **8.7** | +0.6 | 6 处 io.ReadAll 修复，limit 上界补全 |
| 架构 | 7.8 | **8.3** | +0.5 | R1-R6 重构全部生效 |

**综合评分：8.4 / 10（上次 7.9，+0.5）**

### 最高优先级行动项（Top 3）

1. **[P0] scheduler + stdio goroutine panic recovery** — 架构评估和安全评估同时发现。scheduler 的任务 goroutine 和 MCP stdio 的转发 goroutine 各缺一个 `defer recover()`。修复成本极低（各加 5 行），但不修复会导致任务槽位死锁或 MCP stdio 进程崩溃。

2. **[P1] SearchText/SearchTextFiltered 加 limit 上界 + entity_relations UNIQUE 约束** — 数据库和 AI 评估分别发现。Store 层的全文检索方法无上界保护，绕过 Retriever 层可触发大查询。entity_relations 表缺 UNIQUE 约束导致并发实体抽取可创建重复关系。两项均为一行修改。

3. **[P1] Context 读接口 scope 过滤 + Consolidator 删除 fallback** — 安全和 AI 评估分别发现。Context Get/ListChildren/ListSubtree 跳过归属检查可越权读取。Consolidator `creator == nil` fallback 路径可绕过 Manager 层写入。均为功能正确性问题。
