## 数据库评估报告

### 总体评分：8.1 / 10

这是一个设计成熟、层次分明的 SQLite 嵌入式数据库方案。核心机制（FTS5 同步、事务安全、连接池配置、scanDest 模式）均已到位。主要扣分点集中在：fresh schema 与 latestVersion 的版本不同步漏洞、UpdateMetadata 的 TOCTOU 竞争、OFFSET 分页在大表下的性能退化，以及几处缺失索引。

### 关键发现

#### CRITICAL

**C1 — freshSchema 版本滞后于 latestVersion（新库跳过 V23 迁移）**

`sqlite_migration.go:14` 定义 `latestVersion = 23`，但 `sqlite_schema.go:436` 的 `createFreshSchema` 只写入版本 22，且没有 `candidate_for` 列的索引 `idx_memories_candidate_for`。

新库走 `createFreshSchema` 路径后，`getCurrentVersion` 返回 22，然后 `version >= latestVersion`(23) 为 false，将再次执行 `migrateV22ToV23`，但此时表已有 `candidate_for` 列，`ALTER TABLE` 会因 `IsColumnExistsError` 跳过，版本号才被写成 23。这意味着新库需要经历一次多余的迁移循环，且存在 `createFreshSchema` 与增量迁移的行为差异窗口。

修复方案：`createFreshSchema` 应记录版本 23，并在索引定义中补充 `idx_memories_candidate_for`。

**C2 — UpdateMetadata 存在 TOCTOU 竞争条件**

`sqlite_session.go` 的 `UpdateMetadata` 方法执行 read-modify-write 模式，但没有事务包裹。两个并发调用在读取之后、写回之前可以互相覆盖彼方的修改。SQLite WAL 模式下写操作串行，但读不阻写，依然存在丢失更新的风险。

修复方案：将三步操作包入 `db.BeginTx` 事务，或改用单条 `UPDATE ... SET metadata = json_patch(metadata, ?)` 的原子写。

#### HIGH

**H1 — OFFSET 分页在大表上性能退化**

`List`、`ListByContext`、`ListByContextOrdered`、`ListBySourceRef` 均使用 `LIMIT ? OFFSET ?` 模式。当 memories 表达到数万行时，`OFFSET 5000` 会迫使 SQLite 跳过 5000 行。

修复方案：对高频搜索接口改用游标分页：`WHERE updated_at < ? AND id < ? ORDER BY updated_at DESC, id DESC LIMIT ?`。

**H2 — memories 表缺少 `candidate_for` 部分索引（freshSchema 路径）**

`sqlite_schema.go` 的 `createFreshSchema` 索引列表中没有 `idx_memories_candidate_for`，而 `ListCandidates` 查询在没有索引的情况下将对全表进行扫描。

**H3 — FTS5 external-content 模式下 Restore 操作可能触发唯一约束冲突**

软删除将 `deleted_at` 设为非 NULL，从而将行移出 `content_hash` 唯一索引过滤范围。当同一 `content_hash` 的记忆被软删除后再次 Create，随后 Restore 会导致两条记录冲突。

**H4 — `ListMissingExcerpt` 缺少 excerpt 列索引**

heartbeat 定期调用此查询将触发全表扫描。

**H5 — `UpdateState` 使用 `datetime('now')` 而非应用层时间**

与其他写操作（Create、Touch）使用应用层 UTC 时间不一致，在时区不一致或测试环境中产生难以追踪的问题。

#### MEDIUM

- M1: `syncFTS5Tx` 在 `CreateBatch` 中 FTS5 sync 失败仅 Warn 而非 Error
- M2: `visibilityCondition` 通过 `fmt.Sprintf` 拼接条件字符串，维护时容易引入错误
- M3: `PurgeDeleted` FTS5 清理阶段仍将所有 rowid 收集到内存中再逐条删除
- M4: `mmap_size` 通过单独 `db.Exec` 设置而非 DSN 参数
- M5: `ListTimeline` 使用 `COALESCE(happened_at, created_at)` 表达式排序无法利用索引

#### LOW

- L1: `idx_contexts_path` 与 `UNIQUE` 约束重复
- L2: `team_id NOT NULL DEFAULT ''` 与业务语义不符
- L3: 迁移文件 V16→V20 文件命名与实际内容范围歧义
- L4: `Reinforce` 操作不在事务内（单条 UPDATE 在 WAL 下是原子的，但与其他写操作风格不一致）
- L5: `idempotency_keys` 表没有过期清理的定时机制

### 改进建议（按优先级排序）

1. **P0** — 修复 freshSchema 版本滞后（版本写 23，补 `idx_memories_candidate_for`）
2. **P1** — 修复 UpdateMetadata 竞争条件（事务包裹或原子 JSON merge）
3. **P2** — 补充缺失索引（`missing_excerpt`、`candidate_for`、`ListTimeline`）
4. **P3** — 修复 UpdateState 时间来源一致性
5. **P4** — 高频列表接口引入游标分页
6. **P5** — 删除 `idx_contexts_path` 冗余索引
7. **P6** — 统一迁移文件命名规范

### 亮点

1. **双轨迁移架构**：新库走 `createFreshSchema` 一步到位，老库走增量迁移，幂等性良好
2. **scanDest 三件套执行彻底**：36 列统一模式，消除手动 Scan 重复代码
3. **FTS5 事务原子性**：所有写操作在同一事务内完成 FTS5 同步
4. **连接池 + PRAGMA DSN 配置正确**：每个新连接继承关键设置
5. **错误分类集中化**：`store/errors.go` 统一导出错误判断
6. **`PurgeDeleted` 子查询避免参数爆炸**
7. **部分索引（Partial Indexes）广泛使用**：`WHERE deleted_at IS NULL` 等过滤显著减小索引体积
8. **FTS5 查询使用 CTE 消除 `bm25()` 重复计算**

## AI/检索评估报告

### 总体评分：7.5 / 10

LocalMem 的检索系统在架构设计上展现出较高的工程成熟度，三路检索 + 加权 RRF 融合 + 多层后处理管线构成了一个功能完备的 retrieval pipeline。代码质量整体良好，注释覆盖充分，错误处理规范，并发安全考虑到位。

### 关键发现

#### CRITICAL

**C1 — Qdrant 向量去重使用 nil identity 绕过可见性过滤**

`memory/dedup.go:89` 中 `checkVectorDedup` 使用 `nil` identity 调用 `vecStore.Search`，绕过了可见性过滤。跨团队、跨用户的私有记忆都会被纳入去重候选。用户 A 的私有记忆可能导致用户 B 的相似记忆被错误判定为重复。

**C2 — Qdrant SearchFiltered 的 Must + Should 语义不一致**

`store/qdrant.go:238-244` 中，当 `SearchFiltered` 没有 identity 信息时，不做可见性限制，与 `Search` 方法（nil identity 时仅保留 public）的行为不一致。

#### HIGH

**H1 — Embedding 缓存 LRU 的读-写竞争窗口**

`embed/cache.go:53-61` 在 RLock 释放到 Lock 获取之间存在竞争窗口。另一个 goroutine 可能在此间隙中驱逐该 elem，导致链表结构损坏。

**H2 — 强度衰减模型缺乏时间零点校准和最低下限**

`pkg/scoring/strength.go:13-31` 没有最低强度下限，长期未访问的记忆 strength 趋近于 0，即使内容高度相关也无法被检索到。

**H3 — 图谱检索实体级联扇出未限制**

图谱遍历的每层 `GetEntityRelations` 返回的关系数量没有上限。高连接度实体在 depth=2 时可能指数级扇出。

**H4 — RemoteReranker 对 results 做原地修改**

`search/reranker_remote.go:239` 直接修改传入的 `results` slice 中的 Score，违反不可变原则。

#### MEDIUM

- M1: HyDE 生成与主检索流串行执行（可并行化）
- M2: 查询预处理的意图分类规则脆弱（纯正则 + 长度阈值）
- M3: 自适应重试策略过于简单（硬编码阈值 0.3）
- M4: Embedding 缓存缺乏模型版本隔离（key 不含模型名称）
- M5: Reflect 引擎 token budget 计算不精确
- M6: Core memory 固定分数 2.0 可能破坏排序语义

#### LOW

- L1: RRF 单路直返跳过分数归一化
- L2: OpenAI Embedder 硬编码 API URL
- L3: Embedding batch 全量未命中时重复 hash 计算
- L4: 熔断器 half-open 状态无并发限制

### 改进建议（按优先级排序）

1. **[CRITICAL]** 修复向量去重的数据隔离漏洞 — 传入当前用户 identity
2. **[CRITICAL]** 统一 SearchFiltered 的可见性行为
3. **[HIGH]** 修复 LRU 缓存的竞争条件 — 改为写锁或 `sync.Map`
4. **[HIGH]** 为强度衰减设置最低下限（如 0.05）
5. **[HIGH]** 限制图谱遍历的实体扇出（maxVisitedEntities=50）
6. **[MEDIUM]** Embedding 缓存 key 加入模型标识
7. **[MEDIUM]** HyDE 搜索并行化
8. **[MEDIUM]** OpenAI Embedder 支持自定义 base URL

### 亮点

1. **三路并行检索架构设计精良** — FTS5/Qdrant/Graph + HyDE，每通道独立 goroutine + panic recovery
2. **查询预处理管线功能丰富** — 分词、停用词、同义词、实体匹配、意图分类、LLM 增强、HyDE 生成
3. **多层后处理管线** — Backfill -> Rerank -> Kind/Class Weighting -> Scope Priority -> Strength Weighting -> MMR -> Adaptive Retry -> Core Injection
4. **Reflect 引擎自适应 Top-K** — 根据轮次阶段和 token 消耗动态调整检索数量
5. **Extractor 实体规范化两阶段策略** — 精确匹配 + LLM 模糊匹配
6. **远程 Reranker 熔断器保护** — 三态熔断 + 优雅降级
7. **记忆强度模型** — 指数衰减 + 访问频率对数加成 + tier 分级
8. **RRF 实现稳定排序** — 同分字典序 + 浮点容差

## 安全评估报告

### 总体评分：7.2 / 10

整体安全基线良好：认证、SQL 注入防护、路径遍历防护均已实现；常见低级错误已得到处理。主要风险集中在身份边界模糊、MCP 无鉴权时全开放、外部服务通信无 TLS 验证。

### 关键发现

#### CRITICAL

**C1 — MCP SSE 服务器在无 `api_token` 时完全开放**

`mcp/server.go:42-48` 中 `checkAuth` 在 `cfg.APIToken == ""` 时直接返回 `true`。`api_token` 默认值为空字符串。任何能访问 8081 端口的客户端可无限制调用所有 MCP 工具，读写任意 TeamID 下的记忆数据。

**C2 — `X-User-ID` Header 可被客户端任意伪造，直接决定数据归属**

`api/middleware.go:65-70` 中 `OwnerID` 完全来自客户端提交的 `X-User-ID` Header。任何持有合法 API Key 的用户可以传入他人的 `ownerID`，访问对方的私有记忆。根本原因：API Key 只解析出 `TeamID`，`OwnerID` 未绑定在 API Key 上。

#### HIGH

- H1: Docker Compose 内部服务（Tika、Docling、Qdrant）无认证，端口对外暴露
- H2: FTS5 查询中 `visCond` 字符串拼接模式（当前不可利用，但是结构性风险）
- H3: 外部 HTTP 服务使用明文通信且无 TLS 校验配置项
- H4: `isAllowedMIME` 白名单包含 `application/zip` 过宽
- H5: MCP stdio 模式无任何鉴权

#### MEDIUM

- M1: `X-User-ID` 无长度/格式/字符集限制
- M2: panic 信息写入 `error_msg` 并返回给调用者
- M3: 认证关闭时固定 `team_id: "default"`
- M4: CORS 配置在认证禁用时接受通配符 `*`
- M5: 对话摄取单条消息内容无大小上限
- M6: FTS5 sanitize 未处理 `NEAR/N` 语法
- M7: 文档上传 duplicate check 存在 TOCTOU 竞态
- M8: `RetainTool` 接口签名可能导致未来误用

#### LOW

- L1: 速率限制基于 `ClientIP()` 可被伪造 `X-Forwarded-For` 绕过
- L2: limiter map 清理逻辑不够及时
- L3: `auth.api_keys[].key` 不支持 `${ENV_VAR}` 展开
- L4: MCP SSE CORS 默认值 `http://localhost:*` 非标准
- L5: 文件存储目录权限 `0755` 过宽
- L6: Logger 可能记录含敏感参数的 URL

### 改进建议（按优先级排序）

1. **P0** — MCP `api_token` 强制非空校验
2. **P0** — `OwnerID` 绑定到 API Key
3. **P1** — Docker Compose 移除内部服务外部端口
4. **P1** — 外部 HTTP 客户端添加 TLS 配置
5. **P1** — `auth.api_keys[].key` 支持环境变量展开
6. **P1** — `X-User-ID` 格式校验（128 字符，限定字符集）
7. **P2** — panic 信息不应返回给客户端
8. **P2** — MIME 白名单精确化（替换 `application/zip`）
9. **P2** — FTS5 sanitize 补充 `NEAR/N` 过滤
10. **P2** — 文件存储目录权限改为 `0700`

### 亮点

1. **全面的参数化查询** — 无字符串拼接用户输入
2. **恒定时间令牌比较** — `crypto/subtle.ConstantTimeCompare` 防时序攻击
3. **完整的安全响应头** — 全套 SecurityHeaders
4. **路径遍历防护** — `filepath.Abs` + `HasPrefix` 双重校验
5. **响应体大小限制** — 所有外部 HTTP 使用 `io.LimitReader`
6. **多层速率限制** — API + MCP + LLM 三层独立限速
7. **错误信息标准化** — `mapError()` 统一映射
8. **HTTP 服务器超时全面配置** — 防 Slowloris 攻击

## 架构评估报告

### 总体评分：8.0 / 10

LocalMem/IClude 展现出一个成熟的、经过多轮重构优化的 Go 项目架构。分层清晰、接口设计合理、feature gating 模式一致、错误处理规范。主要扣分项集中在个别文件体量超标、配置管理全局单例残留、以及双写一致性保证偏弱。

### 关键发现

#### CRITICAL

无 CRITICAL 级别问题。项目在安全、数据完整性、依赖流向等关键方面均无阻塞性架构缺陷。

#### HIGH

**H1 — `config.AppConfig` 全局单例与 DI 体系不一致**

`config.go:295` 声明 `var AppConfig Config` 全局单例，任何包可绕过 DI 直接访问配置，破坏可测试性。

**H2 — `memory/manager.go` 达 837 行，`Create()` 方法职责过重**

`Create()` 方法 156 行，内联了去重、标签关联、上下文计数、向量写入、摘要生成、实体抽取排队等 6 个副作用，违反项目自身 Quality Gate Rule #17。

**H3 — `search/retriever.go` 达 895 行，单文件承载过多职责**

包含核心检索、图谱遍历、LLM 实体抽取、时间通道、类型加权、scope 加权、token 裁剪、回填等多种职责。

**H4 — 双写策略无补偿机制**

SQLite 写成功后 Qdrant 失败仅记日志，无补偿任务扫描不一致记录。

#### MEDIUM

- M1: MCP 适配器模式增加不必要间接层（纯透传适配器定义在 main.go）
- M2: `sqlite_memory_lifecycle.go` 达 585 行，逻辑密度高
- M3: Viper 默认值 130+ 行，缺乏结构化验证
- M4: `MemoryStore.DB()` 返回 `interface{}` 打破封装性
- M5: `Update()` 方法 20+ 个逐字段 nil 检查缺乏可扩展性
- M6: 错误分类使用字符串匹配依赖驱动消息格式

#### LOW

- L1: MCP Session Star 提醒硬编码在协议层
- L2: `memoryColumns` 字符串常量手工维护 36 列
- L3: 检索权重全部硬编码
- L4: Logger 全局单例无法在测试中替换

### 改进建议（按优先级排序）

1. **P0** — 增加向量同步补偿任务（2-3h）
2. **P0** — 消除 `config.AppConfig` 全局单例（3-4h）
3. **P1** — 拆分 `manager.go` Create 方法（1-2h）
4. **P1** — 拆分 `retriever.go` 到多文件（1-2h）
5. **P1** — 添加 `Config.Validate()`（2h）
6. **P2** — 移除 `MemoryStore.DB()`（1h）
7. **P2** — MCP 适配器移入独立包（1h）
8. **P3** — 检索权重配置化（1h）

### 亮点

1. **优秀的接口分解与组合模式** — 5 个子接口 + 组合接口，符合 Go 小接口哲学
2. **成熟的 Feature Gating 模式** — 一致 nil check，单一二进制按配置灵活降级
3. **精心设计的并行检索架构** — 4 通道并行 + panic recovery + weighted RRF
4. **bootstrap 包依赖装配设计** — 共享初始化逻辑，装配顺序清晰
5. **规范的错误处理体系** — sentinel errors + `mapError()` + 逐层包装
6. **文件拆分和职责边界** — Store 层 26 文件平均 267 行
7. **MCP Registry 线程安全设计** — `sync.RWMutex` 读写分离
8. **延迟注入解决循环依赖** — `SetCreator()`、`SetQueue()` 模式

### 最高优先级行动项（Top 3）

**1. [安全] MCP 鉴权 + OwnerID 身份绑定**
- MCP 服务在无 `api_token` 时完全开放，任何人可读写所有记忆
- `X-User-ID` 可被伪造，同团队用户可越权访问私有记忆
- 行动：强制 `api_token` 非空校验；将 `OwnerID` 绑定到 API Key 而非 Header

**2. [AI/检索] 向量去重数据隔离 + Embedding 缓存竞争修复**
- 向量去重使用 nil identity 绕过可见性过滤，导致跨用户去重错误
- LRU 缓存在 RLock 升级 Lock 期间存在竞争窗口，高并发下可能 panic
- 行动：去重传入当前 identity；缓存改为写锁或 `sync.Map`

**3. [数据库+架构] freshSchema 版本同步 + 双写补偿机制**
- 新库 freshSchema 写入版本 22 但 latestVersion 为 23，导致多余迁移循环
- SQLite/Qdrant 双写无补偿，Qdrant 临时不可用时产生数据不一致
- 行动：freshSchema 同步到 V23 + 补索引；scheduler 增加 vector-sync 补偿任务
