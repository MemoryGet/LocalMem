## CRITICAL 问题

### C-1: 迁移函数 V3→V4 / V4→V5 缺少版本号更新，导致重复迁移风险

**文件**: `internal/store/sqlite_migration.go`，第 81-92 行

```go
// V3→V4
if version < 4 {
    if err := migrateV3ToV4(db, tok); err != nil { ... }
    // ← 缺少 version = 4
}

// V4→V5
if version < 5 {
    if err := migrateV4ToV5(db); err != nil { ... }
    // ← 缺少 version = 5
}
```

**影响**: 当 V3→V4 内部部分成功但最终 Commit 失败（schema_version 未写入），下次重启 version 仍为 3，V3→V4 会重新执行 `DROP TABLE IF EXISTS memories_fts` + 重建，导致已有 FTS5 数据被清空，且全量记忆重新分词。对比 V1→V3 以及 V5→V8 均正确赋值了 `version = N`。

### C-2: Delete() FTS5 删除与主表删除非原子（上次评估仍未修复）

**文件**: `internal/store/sqlite.go`，第 394-424 行

`Delete()` 先在事务外调用 `deleteFTS5ByRowID()`，再执行 `DELETE FROM memories`。两个操作之间如果进程崩溃，FTS5 已删但主表记录仍在，产生幽灵索引。对比 `Update()` 已正确使用 `deleteFTS5ByRowIDTx` + `syncFTS5Tx` 在同一事务内完成。

## HIGH 问题

### H-1: Restore() 未重建 FTS5 索引（新发现）

**文件**: `internal/store/sqlite.go`，第 697-714 行

`SoftDelete()` 在第 691 行删除了 FTS5 条目（best-effort），但 `Restore()` 完全没有重建对应的 FTS5 行。恢复后的记忆对全文搜索不可见，直到下次 Update 或迁移重建触发同步。

### H-2: OFFSET 分页在大表上性能退化（上次评估仍未修复）

**文件**: `internal/store/sqlite.go`，第 436、513、1139 行；`internal/store/sqlite_document.go`，第 123-124 行

`List()`、`ListByContext()`、`ListByContextOrdered()`、`DocumentStore.List()` 均使用 `LIMIT ? OFFSET ?`。当 offset 值较大时 SQLite 必须扫描并丢弃所有前置行，时间复杂度 O(offset)。推荐改为游标分页：`WHERE updated_at < ? AND id < ? ORDER BY updated_at DESC LIMIT ?`。

### H-3: Get() 每次调用都执行无条件写操作（上次评估仍未修复）

**文件**: `internal/store/sqlite.go`，第 280-299 行

每次 `Get()` 都触发 `UPDATE memories SET access_count = access_count + 1`，批量读取场景变成写放大。接口 `IncrementAccessCount` 已存在（第 741 行），`GetVisible()` 无此问题（第 302-317 行）。

### H-4: CleanupExpired() 软删除后不清理 FTS5（新发现）

**文件**: `internal/store/sqlite.go`，第 1041-1056 行

批量 UPDATE 设置 `deleted_at`，但未清理受影响行的 FTS5 索引。与 `SoftDelete()` 不一致（后者有 best-effort FTS5 清理）。过期记忆的 FTS5 条目仍存在，会污染 BM25 排名结果。

### H-5: DeleteTag() / DeleteEntity() 非原子级联删除（上次评估仍未修复）

**文件**: `internal/store/sqlite_tags.go`，第 92-111 行；`internal/store/sqlite_graph.go`，第 154-179 行

两者均先删除关联表，再删除主记录，但在事务外执行。若第一步成功第二步失败，产生数据不一致。

## MEDIUM 问题

### M-1: (team_id, visibility) 缺少复合索引（上次评估仍未修复）

**文件**: `internal/store/sqlite_migration.go`

`visibilityCondition()` 的核心查询路径需要 `(team_id, visibility)` 复合索引才能走 Index Lookup，目前缺失。

### M-2: 所有关联表缺少 FOREIGN KEY 约束

**文件**: `internal/store/sqlite_migration.go`，第 261-307 行

`memory_tags`、`memory_entities`、`entity_relations` 均未声明 `FOREIGN KEY ... REFERENCES`，尽管 `PRAGMA foreign_keys=ON` 已开启。只靠应用层手动级联删除。

### M-3: contexts.memory_count 没有 CHECK 约束防止负数

`DecrementMemoryCount()` 执行 `memory_count = memory_count - 1`，无 `CHECK (memory_count >= 0)` 约束。

### M-4: Queue.Poll() SELECT 与 UPDATE 之间存在竞争窗口

**文件**: `internal/queue/queue.go`，第 98-163 行

SQLite 单写者模型下低风险，但未来多 Worker 场景需改为 CAS 模式。应在代码中注释说明限制。

### M-5: Queue 与 Migration 对 async_tasks 表存在双重建表逻辑

**文件**: `internal/queue/queue.go`，第 45-70 行；`internal/store/sqlite_migration.go`，第 693-730 行

两处完全相同的 DDL，未来变更时易漏改。

### M-6: 迁移 V3→V4 在大数据集下无 batch 处理

全量 FTS5 重建和 content_hash 回填在同一个事务内完成。数万条记忆时持续数十秒，期间所有写操作被阻塞。

## LOW 问题

### L-1: 迁移 V1→V2 中 FTS5 创建使用 CREATE VIRTUAL TABLE（无 IF NOT EXISTS）

防御性最佳实践建议加上 `IF NOT EXISTS`。

### L-2: DocumentStore.Create() 在 INSERT 前执行了额外的 COUNT(*) 查询

**文件**: `internal/store/sqlite_document.go`，第 77-83 行

存在 TOCTOU。第 98-100 行已有 UNIQUE constraint 错误捕获，前置 COUNT 查询冗余。

### L-3: Pool 配置 MaxOpenConns=5，多并发 Goroutine 场景可能不足

heartbeat + scheduler + queue worker 同时持有长事务时容易达到上限。

### L-4: 缺少 cache_size 和 synchronous PRAGMA 配置

WAL 模式下 `synchronous=NORMAL` 可将写延迟降低约 30%，`cache_size` 默认仅 2MB 建议提高。

## 亮点

1. **FTS5 原子同步已大量修复**: Create/Update/CreateBatch/PurgeDeleted 均在同一事务内完成 FTS5 同步
2. **迁移系统设计严谨**: 所有迁移函数均包含 `defer tx.Rollback()`，幂等 `CREATE IF NOT EXISTS`，正确忽略 "duplicate column name" 错误
3. **visibilityCondition 封装良好**: 统一处理多租户可见性，正确区分 TeamID 为空的向后兼容
4. **sqlbuilder 避免字符串拼接**: 参数化彻底，零 SQL 注入风险
5. **Queue.Poll() 事务原子化**: SELECT + UPDATE 在同一事务内防止重复消费
6. **编译期接口检查**: 每个 store 实现都有 `var _ Interface = (*Impl)(nil)`
7. **V6→V7 索引迁移已完成**: entity_relations 和 memory_entities 索引缺失已修复
