## 数据库评估

### 总体评分：7.5 / 10

总体设计成熟、工程质量较高。FTS5 架构合理，迁移系统具有良好的幂等性保护，事务边界清晰，连接池配置合理。主要扣分点集中在：`PurgeDeleted` 遗漏了 `memory_derivations` 的级联清理（潜在孤儿数据）、`OFFSET` 分页在大表上的性能风险、以及 `freshSchema` 与 `latestVersion` 的版本号注释不一致。

### 关键发现

#### CRITICAL

**C-1: `PurgeDeleted` 未清理 `memory_derivations` — 孤儿行风险**

`sqlite_memory_lifecycle.go` 第 359-372 行的 `PurgeDeleted` 函数在硬删除 `memories` 记录之前，手动清理了 `memory_tags` 和 `memory_entities`，但完全遗漏了 `memory_derivations`。虽然 `memory_derivations` 的 FK 定义了 `ON DELETE CASCADE`，但在 Go 连接池复用连接场景下 CASCADE 可能静默失效。建议在 `PurgeDeleted` 中显式删除 `memory_derivations` 行。

**C-2: `freshSchema` 版本注释与实际版本号不一致**

`sqlite_schema.go` 第 13 行注释写"Create final V16 schema"，但实际插入版本号是 17。日志打印"fresh schema V17 created"，注释却仍指向 V16，容易使维护者误判当前最终态。

#### HIGH

**H-1: `OFFSET` 分页在大记忆表上的线性扫描风险**

`List`、`ListByContext`、`ListByContextOrdered`、`ListBySourceRef` 四个方法使用 `LIMIT ? OFFSET ?` 模式。`offset` 没有上界约束，理论上可传入极大偏移。建议改用 cursor 分页。

**H-2: 软删除后 `memory_derivations` 孤儿行不清理**

软删除时 CASCADE 不触发，`memory_derivations` 中的行变成语义上的悬空引用。

**H-3: `entities` 表的 `ListEntities` 无条件时走全表扫描**

缺少 `updated_at` 和 `scope + entity_type` 索引。

**H-4: `visibilityCondition` 的 OR 复合条件无法利用复合索引**

SQLite 对 OR 子句的索引选择能力有限，核心查询往往退化为全扫描。

#### MEDIUM

- M-1: `async_tasks` 表无 Store 接口（死表）
- M-2: `checkTokenizerChange` FTS5 全量重建长事务锁定
- M-3: `consolidated_into` 空字符串与 NULL 语义混用
- M-4: `migrateV15ToV16` 的 `DROP COLUMN` 失败处理过于宽松
- M-5: `PurgeDeleted` 大批量时 IN 子句参数爆炸风险

#### LOW

- L-1: UUIDv4 随机主键不利于 B-Tree 局部性
- L-2: `GetDerivedFrom/To` 缺覆盖索引
- L-3: `ListEntities` 缺 limit 上界
- L-4: `SearchText` 的 `wordCoverage` CJK 假阳性
- L-5: 迁移 v13→v14 重复 PRAGMA table_info

### 亮点

1. FTS5 external content 模式 + 显式事务同步，保证原子一致性
2. BM25 三列加权 + CTE 消除重复计算
3. 双轨迁移（freshSchema + 增量），新库跳过冗长迁移链
4. `scanDest` + `scanFields()` + `toMemory()` 三件套消除重复 Scan
5. WAL + NORMAL synchronous + mmap 256MB 连接池配置合理

## AI/检索评估

### 总体评分：7.8 / 10

成熟度较高的记忆检索系统，三路检索管道 + 加权 RRF 融合 + 多层精排在同类项目中属于上游水平。主要扣分集中在并发执行缺失、RRF 参数静态化、以及记忆演化自动触发尚未完全闭环。

### 关键发现

#### CRITICAL

**C1. 三路检索串行执行，延迟叠加**

`retriever.go` 中 FTS5、Qdrant、Graph 三个通道严格串行执行。Hybrid 模式下延迟叠加可达 10+s。三通道无数据依赖，可用 `errgroup` 并行化，将延迟降至 max(FTS, Qdrant, Graph)。

**C2. Reflect 引擎 evidenceTokens 变量遮蔽导致预算计算失真**

`engine.go` 第 334-335 行重新声明了与外层同名的 `evidenceTokens` 局部变量，导致 priorRoundSummary 中 `EvidenceTokens` 字段赋值使用了错误的变量。应重命名消除遮蔽。

#### HIGH

- H1: HyDE 通道走 FTS5 搜索而非向量检索，偏离 HyDE 论文核心设计
- H2: 图谱通道 N+1 查询问题（O(FTS_top * entities * depth * relations) 次 SQLite 查询）
- H3: OverlapReranker 缺跨语言能力（同义词/缩写/形态变化无法匹配）
- H4: `ApplyStrengthWeighting` 直接 mutate 输入 SearchResult.Score（违反不可变性原则）

#### MEDIUM

- M1: RRF k=60 硬编码，缺自适应机制
- M2: 意图分类纯规则驱动，不支持多意图叠加
- M3: SQLite-only 模式无 MMR 多样性
- M4: 自适应重试可能导致双倍延迟
- M5: Reflect 缺信息增益检测和提前终止
- M6: 时间通道权重硬编码 1.2
- M7: semantic → procedural 自动演化路径缺失

#### LOW

- L1: Graph 深度衰减公式过于陡峭
- L2: 熔断器缺 metrics 暴露
- L3: OverlapReranker 和 RemoteReranker 的 ScoreWeight 语义不同
- L4: ExperienceRecaller 变量名语义不匹配

### 亮点

1. 查询预处理管道：五步流水线 + 意图驱动权重 + HyDE 生成
2. 加权 RRF 融合超越标准等权 RRF
3. RemoteReranker + 熔断器优雅降级
4. adaptiveTopK 综合轮次/预算/证据质量三维动态调整
5. 记忆强度衰减模型：五级保留 + 指数衰减 + 对数访问加成
6. 溯源链完整性：双向遍历 + maxLineageDepth 防环
7. LLM 输出 4 级 fallback 解析

## 安全评估

### 总体评分：7.5 / 10

整体安全姿态良好：认证体系完整、参数化查询覆盖全面、输入边界防护到位、敏感数据无硬编码。无 CRITICAL 级问题。

### 关键发现

#### CRITICAL

无。

#### HIGH

**H-1: test-dashboard `suite` 参数未验证，可绕过到命令注入**

WebSocket 消息中的 `msg["suite"]` 直接插入 `exec.CommandContext` 路径参数，无白名单校验。WebSocket 接受所有 Origin。

#### MEDIUM

- M-1: MCP 服务器 CORS 默认允许所有来源（`*`）
- M-2: HTTP 主服务器缺 `IdleTimeout` 和 `ReadHeaderTimeout`（Slowloris 风险）
- M-3: `isAllowedMIME` 中 `application/zip` 映射过宽
- M-4: 文档处理链中 `TeamID` 未注入内存块（跨团队泄露风险）
- M-5: `ListDocuments` 缺 limit 上界
- M-6: MCP stdio 模式无认证机制
- M-7: 错误消息可能泄露内部业务细节

#### LOW

- L-1: `sqlbuilder` 列名/表名字段未参数化（当前安全但依赖开发者约定）
- L-2: CLI hook 裸 `io.ReadAll` 无大小限制
- L-3: rate limiter 可通过 `X-Forwarded-For` 绕过
- L-4: 配置模板缺 `api_keys` 示例

### 亮点

1. 恒定时间比较（`crypto/subtle.ConstantTimeCompare`）防时序攻击
2. 系统身份保护：阻止客户端通过 `X-User-ID: __system__` 自提权
3. 全面的 HTTP 安全响应头（7 个关键头）
4. 外部 HTTP 响应均使用 `io.LimitReader` 防 OOM
5. 文件路径遍历防护：`filepath.Abs` + `strings.HasPrefix` 双重检查
6. 请求体大小限制：全局 4MB + 端点独立限制
7. Magic bytes MIME 检测 + 扩展名白名单双重验证
8. SQL 注入防护：全参数化查询
9. 无硬编码密钥
10. 分层限流覆盖全路由

## 架构评估

### 总体评分：7.5 / 10

分层架构清晰、接口拆分合理、配置管理灵活。主要扣分在于 `search` -> `memory` 的反向依赖、个别大文件超标、以及 MCP 与 API 层的适配器冗余。

### 关键发现

#### CRITICAL

无。

#### HIGH

**H1. `search` 包对 `memory` 包的反向依赖**

`search` 导入 `memory`（`ApplyStrengthWeighting`）违反 CLAUDE.md 声明的单向依赖流。应提取至 `pkg/scoring/`。

**H2. `sqlite_memory_lifecycle.go` 混合生命周期和搜索逻辑**

506 行且混合了 `MemoryLifecycle` 和 `MemorySearch` 两个接口的实现，应拆分。

**H3. MCP `MemoryCreator` 接口签名与 Manager 不匹配**

需手工适配器逐字段映射，当前适配器已缺失 `AutoExtract`、`Tags`、`DerivedFrom`、`MemoryClass` 等字段。

#### MEDIUM

- M1: `resolveEmbedding` 在 Manager 和 Retriever 中重复实现
- M2: 全局单例 config 与依赖注入混合模式
- M3: `Consolidator.SetCreator` 延迟注入打破构造完整性
- M4: MCP 工具层 8 个独立小接口与 Manager 阻抗失配
- M5: `deleteFTS5ByRowID` 和 `deleteFTS5ByRowIDTx` 代码重复
- M6: `Manager.Delete` 与 `Manager.SoftDelete` 行为差异不在方法名中体现

#### LOW

- L1: `store/errors.go` 用 `strings.Contains` 而非结构化错误类型
- L2: limit 上界不一致（100 vs 200）
- L3: `buildVectorPayload` 缺 `owner_id` 和 `visibility`
- L4: `asyncGenerateExcerpt` 已死代码化

### 亮点

1. 接口拆分：`MemoryStore` = 5 子接口组合，MCP 消费侧最小接口定义
2. bootstrap.Init 装配模式干净，cmd/server/main.go 仅 87 行
3. Nil-safe 特性门控贯穿全栈（config → store → manager → router）
4. 双轨迁移，迁移文件按版本范围拆分
5. `scanDest` 三件套统一序列化
6. `pkg/` 提取合理（hashutil, sqlbuilder, tokenutil, tokenizer, qdrant）
7. MCP 与 API 共享业务层，仅协议适配层不同

## 综合总结

### 各维度评分汇总

| 维度 | 评分 | 趋势 |
|------|------|------|
| 数据库 | 7.5 | 稳定 |
| AI/检索 | 7.8 | 上升（B2/B3 刚完成） |
| 安全 | 7.5 | 稳定 |
| 架构 | 7.5 | 稳定 |
| **综合** | **7.6** | — |

### 最高优先级行动项 (Top 3)

1. **[P0] 三路检索并行化** — 使用 `errgroup` 并行执行 FTS5/Qdrant/Graph 通道，预计 P99 延迟降低 40-60%。这是投入产出比最高的单项优化，直接影响用户体感和 LongMemEval 评测耗时。

2. **[P0] PurgeDeleted 补全 memory_derivations 清理** — 在硬删除事务内显式删除 `memory_derivations` 行，不依赖 FK CASCADE，同时将 IN 子句改为子查询或分批处理，防止参数数量超限。

3. **[P1] 消除 search→memory 反向依赖** — 将 `ApplyStrengthWeighting` / `CalculateEffectiveStrength` 提取至 `pkg/scoring/`，恢复严格单向依赖流，为后续模块拆分扫清障碍。
