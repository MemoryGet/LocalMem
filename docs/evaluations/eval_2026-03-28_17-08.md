## 数据库评估

### 总体评分：7.5 / 10

架构选型合理，SQLite + WAL + FTS5 的组合适合单机本地优先场景。迁移体系设计较为严谨，事务边界清晰，参数化查询全覆盖。主要失分集中在：OFFSET 分页未修正、关键查询缺少覆盖索引、`Delete` 操作的 FTS5/主表非原子问题、以及队列轮询的 SKIP LOCKED 缺失。

### 关键发现

#### CRITICAL

**1. `Delete` 操作存在非原子窗口（FTS5 与主表分步删除）**

文件：`internal/store/sqlite.go`，`Delete` 方法

两步操作在事务外串行执行。若主表删除失败，FTS5 索引已被清除，造成数据不一致。对比 `Update` 方法（正确地将 FTS5 操作包在事务内），`Delete` 存在相同问题但未修复。

**2. 队列 `Poll` 轮询不使用 SKIP LOCKED，并发 Worker 时存在锁争用**

文件：`internal/queue/queue.go`，`Poll` 方法

经典的 "SELECT 后 UPDATE" 模式，多 Worker 并发时可能读到同一行。当前使用 `db.BeginTx(ctx, nil)` 等价于 `BEGIN DEFERRED`，存在 TOCTOU 问题。应使用 `BEGIN IMMEDIATE` 或 `UPDATE ... RETURNING`。

#### HIGH

**3. List/ListByContext/ListTimeline 使用 OFFSET 分页，大表下线性退化**

所有列表接口均使用 `LIMIT ? OFFSET ?`，应改为游标分页。

**4. `Get` 方法在读取后无条件执行写操作（访问计数）**

每次 `Get` 都触发写事务，产生额外 WAL 写入和 checkpoint 压力。建议将访问计数更新剥离为异步批量刷新。

**5. `entity_relations` 和 `memory_entities` 缺少外键约束**

`PRAGMA foreign_keys=ON` 已启用，但关联表未声明 `REFERENCES`，孤儿记录只能靠 heartbeat 发现。

**6. 可见性过滤缺乏 `(team_id, visibility)` 复合索引**

#### MEDIUM

**7. `migrateAddPerformanceIndexes` 每次启动都执行（无版本保护）**
**8. `ListSubtree` 的 `LIKE ? || '/%'` 路径前缀扫描效率不确定**
**9. `SearchTextFiltered` 中 BM25 权重参数绑定顺序需验证**
**10. `DeleteEntity` 三步删除不在单一事务内**
**11. `PurgeDeleted` 的 FTS5 清理在主事务外执行**
**12. Queue `Poll` 手动解析时间字符串，未用原生 `time.Time` 扫描**

#### LOW

**13. `memory_tags` 的 `tag_id` 方向缺少独立索引**
**14. `contexts.memory_count` 冗余计数列缺乏一致性保证**
**15. `sqlbuilder.SelectBuilder` 不支持 OFFSET**

### 改进建议（按优先级排序）

| 优先级 | 建议 |
|--------|------|
| P0 | 将 `Delete` 的 FTS5 清理合并进同一事务 |
| P0 | `DeleteEntity`/`DeleteTag` 的多步删除包在事务内 |
| P0 | `queue.Poll` 改为 `BEGIN IMMEDIATE` 确保互斥 |
| P1 | 游标分页替换 OFFSET 分页 |
| P1 | `Get` 访问计数异步批量刷新 |
| P1 | 关联表添加 REFERENCES 外键声明 |
| P2 | `migrateAddPerformanceIndexes` 纳入 V9 版本迁移 |
| P2 | `(team_id, visibility)` 复合索引 |

### 亮点

1. WAL + PRAGMAs 配置规范，连接池参数合理
2. FTS5 external content 模式 + 事务内同步（Create/Update）
3. 迁移体系严谨，8 个版本全部事务包裹、幂等保护
4. 部分索引（Partial Indexes）大量使用，是 SQLite 最佳实践
5. 内容哈希去重，数据库层防止重复写入
6. 队列 `ResetStale` + `Fail` 重试机制完善
7. 参数化查询全覆盖，零 SQL 注入风险

## AI/检索评估

### 总体评分：7.5 / 10

三路检索 + 加权 RRF 融合 + MMR 重排 + 记忆强度衰减的组合在同类项目中相当完备。代码组织清晰，降级策略多层覆盖，nil-safe 设计贯穿始终。主要改进空间在向量检索质量保障、并发安全和可扩展性方面。

### 关键发现

#### CRITICAL

**1. 图谱实体匹配硬上限 500，超出后静默丢失**

`ListEntities(ctx, scope, "", 500)` 将实体列表硬限制为 500 条，超出后系统不报错但检索质量悄然下降。

**2. LLM 实体提取的 `ListEntities` 全量扫描 — O(N) 匹配**

`llmExtractEntities` 用 `strings.EqualFold` 线性扫描全部实体匹配，数千实体时每次检索触发全量遍历。

#### HIGH

**3. Embedding 适配器缺少重试和速率限制**

两个适配器无指数退避重试、速率限制、metrics 收集。

**4. MMR 重排的 N+1 查询问题**

每次启用 MMR 的检索都对 Qdrant 发起额外的批量点查询。

**5. RRF 融合未保留原始通道分数**

所有结果的 Source 统一标记为 `"hybrid"`，丢失各通道的贡献度信息。

**6. FallbackProvider 缺少断路器模式**

主 provider 持续超时时每次请求都浪费等待时间。

#### MEDIUM

**7. Reflect 引擎 token 预算追踪不精确（未计入检索结果 token）**
**8. 图谱遍历排序使用冒泡排序 O(n^2)**
**9. Extractor 实体/关系类型硬编码不可扩展**
**10. 查询意图分类不支持混合意图**
**11. Qdrant 客户端缺少 API 密钥认证**
**12. Embedding 维度不校验**

#### LOW

**13. `EstimateTokens` 英文 token 估算系数偏简单**
**14. Reflect 引擎 auto-save 未限制触发频率**
**15. OpenAI Embedder URL 硬编码，不支持自定义 baseURL**

### 改进建议（按优先级排序）

| 优先级 | 建议 |
|--------|------|
| P0 | 为 GraphStore 添加 `FindEntityByName` 方法，替代全量扫描 |
| P1 | Embedding 适配器增加重试 + 速率限制 |
| P1 | FallbackProvider 增加断路器 |
| P1 | RRF 融合保留通道元数据 |
| P2 | MMR 向量缓存 |
| P2 | Reflect token 预算修正 |
| P2 | 实体/关系类型配置化 |

### 亮点

1. 三路检索 + 加权 RRF 融合架构设计优秀
2. 查询预处理器的意图感知权重调节（CJK vs 英文阈值）
3. 三级 fallback 解析策略（JSON → 正则 → LLM 重试 → 原始降级）
4. 记忆强度衰减模型有理论支撑
5. 全面的 nil-safe 设计，支持渐进式升级
6. MMR 重排实现正确，缺失向量优雅降级
7. GraphStore 的 LLM fallback 实体提取提升长尾召回
8. Extractor 两阶段实体规范化（精确匹配 + LLM 辅助）

## 安全评估

### 总体评分：7.0 / 10

项目整体安全基础扎实，参数化查询、身份注入保护、错误信息归一化等良好实践。主要风险在于：缺乏速率限制、MCP token 时序侧信道、HTTP Body 无大小限制、部分权限校验不完整。

### 关键发现

#### CRITICAL

**1. API 层完全缺失速率限制**

所有 `/v1/*` 路由无速率限制，`/v1/reflect`、`/v1/memories/:id/extract` 等 LLM 密集型接口可被暴力请求耗尽 API 额度。

**2. REST API 无请求 Body 大小限制**

MCP Server 通过 `http.MaxBytesReader` 限制 1MB，但 Gin REST API 侧无等效限制，存在内存耗尽攻击风险。

#### HIGH

**3. MCP Token 验证存在时序侧信道漏洞**

使用普通 `==` 比较 Bearer token，未使用 `crypto/subtle.ConstantTimeCompare`。

**4. `/maintenance/cleanup` 缺乏权限隔离**

任何持有有效 API Key 的用户均可触发全局清理操作。

**5. `Restore`/`Reinforce` 缺少 OwnerID 授权检查**

对比 `Update`/`Delete` 有归属检查，这两个端点未检查当前用户是否为 memory owner。

**6. MCP Server 默认 CORS 为通配符 `*`**

**7. FTS5 MATCH 查询语法注入（逻辑型）**

用户输入直接作为 FTS5 MATCH 操作数，FTS5 语法支持 `OR`/`NOT`/`NEAR` 操作符，可修改搜索逻辑。

#### MEDIUM

**8. HTTP 安全响应头缺失**（X-Content-Type-Options, X-Frame-Options 等）
**9. 不强制 HTTPS / 无 TLS 配置**
**10. `auth.enabled: false` 时无审计警告**
**11. FTS5 MATCH 语法错误返回 500 而非 400**
**12. `X-User-ID` Header 无格式验证**
**13. 批量查询端点认证关闭时可能暴露其他用户数据**

#### LOW

**14. MCP Server panic 无 Recovery 中间件**
**15. MCP `api_token` 为空时仅 Warn 不拒绝启动**
**16. `document.Upload` 的 `FilePath` 字段未验证路径**
**17. `auth.enabled: true` 且 `api_keys` 为空时无 Fatal 提示**

### 改进建议（按优先级排序）

| 优先级 | 建议 |
|--------|------|
| P0 | 添加速率限制（LLM 密集型接口优先） |
| P0 | REST API 添加 Body 大小限制 |
| P0 | MCP token 改用 `crypto/subtle.ConstantTimeCompare` |
| P1 | `Restore`/`Reinforce` 补充 OwnerID 授权检查 |
| P1 | `maintenance/cleanup` 添加系统级权限检查 |
| P1 | FTS5 MATCH 输入清洗（包裹为 phrase query） |
| P2 | 添加安全响应头中间件 |
| P2 | 部署文档要求 TLS 终止 |

### 亮点

1. SQL 注入防护出色，参数化查询全覆盖
2. 系统身份保护正确过滤 `__system__` 冒充
3. 错误信息归一化，不泄露内部细节
4. 可见性模型 private/team/public 三级隔离完善
5. 身份字段强制覆盖，防止客户端伪造归属
6. 纯 Go SQLite 实现（无 CGo 攻击面）
7. `gin.SetMode(gin.ReleaseMode)` 已设置

## 架构评估

### 总体评分：7.5 / 10

项目整体架构设计合理，分层清晰，依赖流向正确，feature gating 机制成熟。在中等规模 Go 项目中属于高于平均水平的架构质量。

### 关键发现

#### CRITICAL

**1. `config.GetConfig()` 全局单例在业务层散射调用**

`Manager.Create()`、`asyncExtract`、`Reinforce`、`heartbeat.Engine`、`consolidation` 直接调用全局配置，违反依赖注入原则，不可测试。

**2. `MemoryStore` 接口过大（25+ methods）**

违反 Go "小接口" 惯例，mock 成本高，`DB() interface{}` 暴露实现细节。

#### HIGH

**3. `DB() interface{}` 泄漏存储实现细节**

调用方做 `DB().(*sql.DB)` 类型断言，破坏接口抽象。

**4. `search` 包直接依赖 `memory` 包**

`memory.ApplyStrengthWeighting()` 应提取到 `model` 或 `scoring` 包，消除潜在循环依赖风险。

**5. `NewManager` 构造函数参数过多（7 个 + variadic）**

参数顺序错误风险高，应使用 Options/Params 模式。

**6. `sqlite.go` 超过 1276 行**

远超项目规定的 1000 行上限，应按职责拆分。

#### MEDIUM

**7. MCP `main.go` 中适配器代码应提取为独立包**
**8. Worker `maxPerTick = 10` 硬编码**
**9. `findOrCreateTag` 全表扫描查找标签**
**10. Heartbeat Engine 运行时重读全局配置**
**11. `IngestConversation` 逐条递增 context memory count**
**12. `model.ErrInvalidInput` 包装方式不一致**

#### LOW

**13. 图谱遍历排序使用冒泡排序**
**14. `config.go` 默认值 85 行连续调用**
**15. `model.Memory` 31+ 字段可考虑嵌套子结构体**

### 改进建议（按优先级排序）

| 优先级 | 建议 |
|--------|------|
| P0 | 配置注入替代全局单例 |
| P0 | 拆分 `MemoryStore` 为 4-5 个小接口 |
| P1 | 提取 `ApplyStrengthWeighting` 到 `model` 包 |
| P1 | 构造函数使用 Params 结构体 |
| P1 | 拆分 `sqlite.go` 为 CRUD/Search/Lifecycle |
| P2 | MCP 适配器提取到独立包 |
| P2 | Worker `maxPerTick` 可配置化 |
| P2 | TagStore 增加 `GetByName` 方法 |

### 亮点

1. 出色的 Bootstrap/Wiring 模式，两个入口共享启动代码
2. 优雅的 Feature Gating（nil 检查，无需 feature flag 框架）
3. 完善的 Best-Effort 双写策略（SQLite 主，Qdrant 降级）
4. Scheduler 的 `atomic.Bool` + CAS 零锁重叠防护
5. 三级检索融合是核心竞争力
6. Queue 的 SQLite 持久化设计符合 local-first 定位
7. 清晰的 sentinel errors 体系
8. 统一的双语注释规范

## 综合总结

### 各维度评分汇总

| 维度 | 评分 | 核心风险 |
|------|------|----------|
| 数据库 | 7.5/10 | Delete 非原子、OFFSET 分页、Queue 并发安全 |
| AI/检索 | 7.5/10 | 实体匹配硬上限、Embedding 无重试、无断路器 |
| 安全 | 7.0/10 | 无速率限制、无 Body 限制、Token 时序侧信道 |
| 架构 | 7.5/10 | 全局配置散射、接口过大、文件过长 |
| **综合** | **7.4/10** | |

### 最高优先级行动项（Top 3）

**1. [安全 + 数据库] 添加 API 速率限制 + Body 大小限制 + MCP Token 安全比较**

这是当前最大的生产安全风险。LLM 密集型接口无速率限制可被恶意请求耗尽 API 额度，REST API 无 Body 限制可导致 OOM，MCP Token 比较存在时序侧信道。建议：
- 引入 `golang.org/x/time/rate`，对 `/v1/reflect`、`/v1/memories/:id/extract` 限速
- Gin 全局中间件 `http.MaxBytesReader` 10MB
- MCP `checkAuth` 改用 `crypto/subtle.ConstantTimeCompare`

**2. [数据库 + 架构] 修复 Delete/DeleteEntity 事务原子性 + Queue Poll 并发安全**

`Delete` 的 FTS5 与主表分步删除、`DeleteEntity`/`DeleteTag` 多步删除不在事务内，以及 Queue `Poll` 的 TOCTOU 问题，三者共同构成数据一致性风险。建议：
- `Delete`/`PurgeDeleted` 的 FTS5 清理合并进同一事务
- `DeleteEntity`/`DeleteTag` 包在事务内
- `queue.Poll` 改为 `BEGIN IMMEDIATE` 事务

**3. [架构] 消除全局配置单例 + 拆分 MemoryStore 接口**

`config.GetConfig()` 在业务层散射调用导致不可测试，`MemoryStore` 25+ 方法的巨型接口违反 Go 惯例。建议：
- 定义 `ManagerConfig` 结构体，通过构造函数注入
- 拆分为 `MemoryReader`/`MemoryWriter`/`MemorySearcher`/`MemoryLifecycle` 小接口
- 提取 `ApplyStrengthWeighting` 到 `model` 包消除 `search → memory` 依赖
