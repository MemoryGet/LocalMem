## 数据库评估报告

### 总体评分: 7.4 / 10

该项目的数据库层体现了良好的工程素养：迁移系统完整有序、扫描结构统一避免重复、FTS5 写入与主表在同一事务内保证一致性、连接池参数配置合理。但存在若干值得修复的中高危问题。

---

### 关键发现

#### CRITICAL

**C1 — 迁移编排器 V9→V12 跳步：`version` 变量未更新**

`/root/LocalMem/internal/store/sqlite_migration.go` 第 128–147 行，迁移控制流中 V1→V9 每步都有 `version = N`，但 V9→V10、V10→V11、V11→V12 三步均未更新本地 `version` 变量。

效果：若数据库当前版本为 9，三步迁移将正确执行（因为 `if version < 10/11/12` 仍成立）。但若某步迁移失败返回 error，下一次重启仍会从 `version=9` 重跑已成功的步骤。虽然迁移函数本身有 `INSERT OR REPLACE INTO schema_version` 以及 `isColumnExistsError` 保护，部分操作是幂等的，但 V9→V10 中对 `documents` 表的 `DROP INDEX` 后紧接 `CREATE UNIQUE INDEX` 在重跑时行为可能不同。

**C2 — `List` / `ListByContext` 使用 OFFSET 分页，无上界保护**

`/root/LocalMem/internal/store/sqlite_memory_read.go` 第 47–88 行，Store 层未设 `limit` 上界（只做下界兜底），API 层对 `/v1/memories` 设了 `limit > 500` 则裁剪，但其他调用路径（内部调用、MCP tools）可绕过此保护。项目 CLAUDE.md 要求所有 `limit` 参数必须有上界（`> 200` 则截断），memory_handler 却设为 500，不统一。

---

#### HIGH

**H1 — 无外键约束，关联表孤儿风险**

全库 DDL 中无一条 `FOREIGN KEY` 声明。虽然 `PRAGMA foreign_keys=ON` 已开启，但开启此 PRAGMA 仅对有 FK 定义的列生效。受影响关联：`memory_tags`、`memory_entities`、`entity_relations`、`memories.context_id`、`memories.document_id`。

**H2 — `memory_handler.go` 的 limit 上界为 500，与项目规范（200）不一致**

**H3 — `syncFTS5`（不带 Tx 的版本）在生产路径中已无调用，但仍保留为代码隐患**

**H4 — `SearchTextFiltered` 中可见性条件可被绕过**

当 `filters.TeamID == "" && filters.OwnerID == ""` 时，查询不附加任何可见性条件，所有记忆均可被返回。

---

#### MEDIUM

- M1: FTS5 external content 模式缺少 `integrity-check` 维护机制
- M2: 多个迁移函数使用 `context.Background()` 而非传入的 context
- M3: `deleteFTS5ByRowIDTx` 操作顺序在 SoftDelete 中容易引起误解
- M4: `ListTags` 无 limit 限制
- M5: `migrateAddPerformanceIndexes` 孤立函数未被调用（死代码）
- M6: `content_hash` 唯一索引与 Restore 的冲突风险

#### LOW

- L1: `PRAGMA synchronous = NORMAL` 在 OS 崩溃时有极小概率丢帧
- L2: `PRAGMA mmap_size=268435456` 硬编码，未从配置读取
- L3: `schema_version` 表 `INSERT OR IGNORE` / `OR REPLACE` 混用

---

### 改进建议（按优先级排序）

1. 修复迁移编排器 V9→V12 的 `version` 更新缺失（C1）
2. 为 `SearchTextFiltered` 补充强制可见性过滤（H4）
3. 统一 limit 上界为 200（H2 + C2 + M4）
4. 为主要外键关联列添加 FK 声明（H1）
5. 删除死代码 `syncFTS5` 和 `migrateAddPerformanceIndexes`（H3 + M5）
6. 添加 FTS5 integrity-check 定期调度（M1）
7. 将迁移改为接受 context 参数（M2）
8. 将 mmap_size 移入配置（L2）

### 亮点

- FTS5 写入与主表在同一事务内，保证原子一致性
- 统一的 `memScanDest` 结构体消除了三份重复扫描逻辑
- 迁移系统幂等性设计（`isColumnExistsError` + `IF NOT EXISTS`）
- BM25 权重可配置 + CTE 消除重复计算
- 分词器变更自动重建 FTS
- 连接池配置合理匹配 WAL 模式

## AI/检索评估报告

### 总体评分: 8.2 / 10

该系统展现了成熟的企业级 AI 检索架构设计，多通道融合、多级 fallback、生命周期管理等核心机制均实现到位。代码质量整体良好，CJK 感知处理、不可变模式、错误包装等方面体现了工程纪律。

---

### 关键发现

#### CRITICAL

**1. asyncExtract goroutine 缺少 panic recovery**

`/root/LocalMem/internal/memory/manager.go` 第 255-265 行，`asyncExtract` 和 `asyncGenerateAbstract` 启动的 goroutine 没有 `defer recover()`，panic 将崩溃整个进程。

**2. OpenAI/Ollama embedding 客户端未限制响应体大小**

`embed/openai.go` 第 149 行, `embed/ollama.go` 第 131 行使用 `io.ReadAll(resp.Body)` 无 `io.LimitReader`。

---

#### HIGH

**3. Extractor 关系去重存在 TOCTOU 竞态** — 先查后写不在同一事务内

**4. LLM Provider 缺少请求级重试** — 对比 embed 层已有 3 次指数退避，LLM 层反而没有

**5. Consolidator 直接调用 memStore.Create 绕过 Manager** — 归纳记忆跳过 embedding + Qdrant 写入

---

#### MEDIUM

- M6: Embedding 缓存 LRU `touchLocked` 是 O(n) 操作
- M7: Token 估算精度（CJK/非CJK 启发式）设计合理但有局限
- M8: RRF 融合后权重叠乘链过长（4 层），`accessBoost` 无上限
- M9: 图谱遍历无总节点数上限
- M10: Reflect 引擎 `priorRounds` 可能无限增长 prompt 长度

#### LOW

- L4: Extractor 实体规范化候选列表策略可优化
- L5: Temporal 通道硬编码权重 1.2 不可配置
- L6: HyDE 文档生成使用 `context.Background()` 脱离请求 context

---

### 改进建议（按优先级排序）

1. [P0] 为所有 async goroutine 添加 panic recovery
2. [P0] 为 embed 客户端添加响应体大小限制
3. [P1] Consolidator 改为通过 Manager.Create 写入
4. [P1] 为 LLM Provider 添加重试机制
5. [P2] Extractor 关系去重改用 UNIQUE 约束
6. [P2] LRU 缓存改用 `container/list` 实现 O(1)
7. [P2] 为 `accessBoost` 添加上限（建议 3.0）
8. [P3] 图谱遍历添加总实体数上限
9. [P3] Reflect 引擎 priorRounds 做滑动窗口

### 亮点

- 三路 RRF 融合架构设计优秀，支持动态权重调整
- 意图感知的查询预处理管道（分词→停用词→同义词→实体→意图→HyDE）
- 3 级 fallback 解析模式（JSON → regex → LLM retry → raw）
- Memory 生命周期管理完备（指数衰减 + 访问频率加成 + 5 级 retention tier）
- Embedding 缓存装饰器设计优雅（`copyVec` 保证隔离）
- Consolidator 层次聚类 + LLM 归纳 + 输出长度校验

## 安全评估报告

### 总体评分: 8.1 / 10

与历史评估对比，本次代码库在认证鉴权和 SQL 注入防护方面有实质性进步。主要失分项集中在外部 HTTP 客户端缺少响应体大小限制（可触发 OOM），以及 limit 上界缺失问题。无 CRITICAL 级别漏洞。

---

### 关键发现

#### CRITICAL

**无**

---

#### HIGH

**H1: Tika/Docling 响应体未限制读取大小（OOM 风险）**

`tika.go` 第 60、64 行和 `docling.go` 第 80 行对外部 HTTP 服务使用无界 `io.ReadAll`。对比 `reranker_remote.go:151` 已正确使用 `io.LimitReader`。

**H2: LLM/Embed/Qdrant 响应体未限制读取大小**

`llm/openai.go` 第 88 行、`embed/openai.go` 第 148 行、`embed/ollama.go` 第 131 行、`pkg/qdrant/client.go` 第 272 行。

**H3: Timeline 端点 limit 参数无上界**

`GET /v1/timeline?limit=10000000` 可触发超大 SQL 查询。

**H4: POST /v1/conversations messages 数组无上界**

可提交数千条消息，单次请求触发等量的 `Create` 调用。

---

#### MEDIUM

- M1: memory_handler List 接口 limit 上界过高（500 vs 规范 200）
- M2: MCP handleMessages goroutine 无 recover
- M3: ContextHandler Get/ListChildren/ListSubtree 无 scope 过滤（读写不对称越权）
- M4: ReflectHandler TokenBudget 无上界校验
- M5: 错误响应在部分路径可能泄露 SQLite 内部状态

#### LOW

- L1: MCP Server 全局速率限制无 per-IP 限流
- L2: context_handler.go 的 ListMemories 方法是死代码
- L3: MCP SSE 连接数上限 100 但无 per-IP 限制

---

### 改进建议（按优先级排序）

1. [P0] 为 tika/docling/llm/embed/qdrant 所有 `io.ReadAll` 加 `io.LimitReader`
2. [P1] Timeline 加 limit 上界（200）
3. [P1] Conversations Ingest 加 messages 最大长度（500）
4. [P1] MCP dispatch goroutine 加 recover
5. [P2] ContextHandler 读操作补充 scope 归属检查
6. [P2] memories limit 上界从 500 降至 200
7. [P2] ReflectHandler 补充 TokenBudget 上界

### 亮点

- SQL 注入防护完整（参数化查询 + FTS5 `sanitizeFTS5Query`）
- 认证鉴权覆盖全面（`ConstantTimeCompare` + `__system__` 冒充阻断）
- 路径遍历防护到位（Save/Get/Delete 三方法全覆盖）
- 安全响应头完整（7 个安全头）
- 输入验证分层（4MB 体限 + 扩展名白名单 + magic bytes）
- 速率限制梯度设计合理（写 20rps / 搜索 10rps / LLM 2rps）

## 架构评估报告 — IClude (LocalMem)

### 总体评分: 7.8 / 10

一个在架构层面经过深思熟虑的 Go 企业级记忆系统。分层清晰、接口拆分合理、特性门控设计成熟。主要扣分项集中在少数并发安全隐患、全局单例耦合、以及 `search` 对 `memory` 的具体类型依赖上。

---

### 关键发现

#### CRITICAL

**C1. 异步 goroutine 缺少 panic recovery (`manager.go` L255, L269)**

`asyncExtract` 和 `asyncGenerateAbstract` 中的 `go func()` 均没有 `defer recover()`，违反项目自身质量关卡规则。

**C2. MCP SSE dispatch goroutine 缺少 panic recovery (`server.go` L182)**

`Dispatch` 处理外部 JSON-RPC 输入，panic 将崩溃进程。

---

#### HIGH

**H1. `search` 包对 `memory` 包的具体类型依赖** — `ApplyStrengthWeighting` 应提取到 `pkg/` 或 `model/`

**H2. 全局配置单例 `AppConfig` 不利于测试和并发安全**

**H3. `sqlite_migration.go` 857 行且持续增长** — 接近 1000 行上限

**H4. `Manager.NewManager` 构造函数参数过多（9 个位置参数 + variadic）** — 应改为 Options 或 Deps 结构体

---

#### MEDIUM

- M1: `config.go` 的 SetDefault 段落过长（100+ 行）
- M2: `Retriever.Retrieve` 方法 ~250 行，复杂度偏高
- M3: `Stores.RawDB` 暴露底层数据库连接
- M4: `MemoryStore.DB()` 返回 `interface{}` 缺乏类型安全
- M5: 环境变量展开仅覆盖部分配置字段
- M6: `AccessTracker.Flush` 使用 `goto` 语句

#### LOW

- L1: MCP main.go 中的 adapter 类型应提取到独立文件
- L2: `kindWeights`/`classWeights` 硬编码在 retriever.go
- L3: 部分 sentinel error 缺少结构化信息

---

### 改进建议（按优先级排序）

1. [CRITICAL] 为所有异步 goroutine 添加 `defer recover()`
2. [HIGH] 将 `ApplyStrengthWeighting` 提取到 `pkg/scoring/` 或 `model/`
3. [HIGH] Manager 构造函数改为 `ManagerDeps` 结构体注入
4. [HIGH] 迁移文件按版本区间拆分
5. [MEDIUM] `config.LoadConfig()` SetDefault 按 section 组织
6. [MEDIUM] 拆分 `Retriever.Retrieve` 为通道子方法
7. [LOW] 检索权重移入配置系统

### 亮点

- 接口拆分精当（MemoryStore → 4 子接口 + 组合接口，零 breaking change）
- `bootstrap.Init()` 集中 wiring + `Deps` 结构体 + cleanup 函数
- nil check 特性门控设计优雅，无需额外 feature flag 框架
- 依赖流向严格单向（实际依赖图是 DAG）
- 调度器 127 行涵盖生产级核心需求
- 质量关卡规则从 14 次实战事故提炼
- 三层配置模板（basic/standard/premium）+ Web 生成器
- 81 个测试文件按层级组织，表驱动风格统一

## 综合总结

### 维度评分汇总

| 维度 | 评分 | 趋势 |
|------|------|------|
| 数据库 | 7.4 / 10 | 迁移系统成熟，FTS5 一致性好，需修复 limit 不统一和可见性绕过 |
| AI/检索 | 8.2 / 10 | 三路 RRF 融合优秀，需修复 goroutine recovery 和 Consolidator 绕过 Manager |
| 安全 | 8.1 / 10 | 认证鉴权和注入防护全面，需补齐外部 HTTP 响应体限制 |
| 架构 | 7.8 / 10 | 分层清晰接口精当，需消除 search→memory 依赖和 Manager 参数膨胀 |

**综合评分: 7.9 / 10**

### 最高优先级行动项（Top 3）

1. **[P0] 异步 goroutine panic recovery** — 4 个评估维度中有 3 个独立发现此问题。`asyncExtract`、`asyncGenerateAbstract`、MCP dispatch 三处裸 goroutine 缺少 `defer recover()`，任何 panic 将崩溃整个进程。修复成本极低（每处加 3 行），但不修复的风险是生产级故障。

2. **[P0] 外部 HTTP 响应体大小限制** — `tika.go`、`docling.go`、`llm/openai.go`、`embed/openai.go`、`embed/ollama.go`、`pkg/qdrant/client.go` 共 6 处 `io.ReadAll` 无 `io.LimitReader`。`reranker_remote.go` 已有正确实现可参考。故障外部服务可触发 OOM。

3. **[P1] Consolidator 绕过 Manager 层写入** — `consolidation.go` 直接调用 `memStore.Create()` 跳过了 embedding 生成和 Qdrant 写入，导致归纳记忆在语义检索中完全不可见。这是功能正确性问题，违反项目"业务写入必须经过 Manager 层"规则。
