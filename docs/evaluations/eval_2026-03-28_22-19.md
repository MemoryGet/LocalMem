## 综合总结

### 最高优先级行动项（Top 3）

**1. [数据库 + AI/检索] SoftDelete/CleanupExpired 事务原子化 + FindEntitiesByName 补索引**

SoftDelete 和 CleanupExpired 的 FTS5 清理仍在事务外，是数据一致性的最后缺口。同时 FindEntitiesByName 缺少 `LOWER(name)` 函数索引，导致改造效果未实际生效。建议：
- SoftDelete 参照 Delete() 使用 BeginTx
- CleanupExpired 三步操作合并至单事务
- 新建 V9 迁移：`CREATE INDEX idx_entities_lower_name ON entities(LOWER(name))`

**2. [安全] REST API Token 恒定时间比较 + 速率限制补全 + Cleanup 权限门控**

AuthMiddleware 的 map 查找存在理论时序攻击面。核心写接口（memories/conversations/documents）缺速率限制。Cleanup 端点任意用户可触发全局清理。建议：
- AuthMiddleware 改用 subtle.ConstantTimeCompare 遍历
- 写接口添加 20rps 限速
- Cleanup 添加 IsSystem() 检查

**3. [架构] 拆分 MemoryStore 接口 + sqlite.go 文件 + Consolidator 配置收窄**

25+ 方法的巨型接口和 1339 行的单文件是最大的可维护性债务。Consolidator 接收完整 Config 违反最小知识原则。建议：
- MemoryStore 拆为 Reader/Writer/Searcher/Lifecycle 四小接口
- sqlite.go 按职责拆为 4 个文件
- Consolidator 改为接收 ConsolidationConfig
