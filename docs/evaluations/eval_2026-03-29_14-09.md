## 数据库评估报告

### 总体评分：7.5 / 10

整体设计成熟，迁移机制严格、FTS5 集成合理、事务边界清晰。主要扣分点：`memory_tags` 缺 `tag_id` 索引、OFFSET 分页、WAL 连接池配置偏保守、`OR ? = ''` 模式阻止索引利用。

### 关键发现

#### CRITICAL
- C1: `memory_tags` 关联表缺少 `tag_id` 索引，`DeleteTag`/`GetMemoriesByTag` 全表扫描
- C2: WAL 模式下 `MaxOpenConns(5)` 无 `_txlock=immediate`，高并发写入触发 `database is locked`

#### HIGH
- H1: 三处 OFFSET 分页，大数据量线性扫描退化
- H2: `PurgeDeleted` 事务外预查询存在 TOCTOU 窗口
- H3: `contexts` 的 `LIKE path || '/%'` 阻止索引前缀优化
- H4: `ListTags`/`documents.List` 使用 `OR ? = ''` 模式阻止索引
- H5: `entity_relations`/`memory_tags` 缺外键约束

#### MEDIUM
- FTS5 未指定 `tokenize=` 参数、`GetEntityRelations` OR 查询低效、死代码 `migrateAddPerformanceIndexes`、`SoftDelete` rowid 获取顺序不一致、`meta` 表 tokenizer 未启动校验、缺 `PRAGMA optimize`、缺 `ConnMaxIdleTime`

#### LOW
- `documents.List` 无默认 limit 上限、`CreateBatch` 时间戳不统一、`contexts.parent_id` 空字符串非 NULL、V10 迁移错误处理不一致

### 亮点
- FTS5 事务内一致性、10 版本幂等迁移链、BM25 CTE 优化、部分索引精准设计、PRAGMA 调优组合合理、批量插入 PreparedStatement、多租户可见性 SQL 抽象

## AI/检索评估报告

### 总体评分：7.5 / 10

三路 RRF 融合架构优秀，意图感知预处理成熟。存在 embedding 输入无截断保护、强度加权与 MMR 执行顺序问题、token 估算精度不足。

### 关键发现

#### CRITICAL
- C1: Embedding 输入无长度截断保护，超长 chunk 导致 API 400 + 3 次无效重试
- C2: `ApplyStrengthWeighting` 在 MMR 之后执行，过期记忆可能在 MMR 阶段被选中后又被过滤，导致返回结果数少于 limit

#### HIGH
- H3: Token 估算精度 +/-15%~33%，Reflect token budget 可能超支
- H4: Chunker overlap 基于字符而非语义边界，`tokensToChars` 对 CJK 高估 3 倍
- H5: Extractor 实体/关系类型硬编码（5+4 种），扩展受限

#### MEDIUM
- RRF 权重通道禁用语义冲突、Reflect 缺多轮上下文累积、`Process()` 手动入口硬编码 TextChunker、图谱 entityLimit 共享导致大量无效查询、Consolidator 候选选取无随机采样

#### LOW
- cosineSimilarity 重复实现、OpenAI Embedder 硬编码 URL、temporal 时间窗口硬编码 7 天

### 亮点
- 三路加权 RRF 融合、意图感知查询预处理、MMR 多样性重排、Reflect 三级 fallback 解析、记忆生命周期管理体系、Best-effort 双写、异步 AccessTracker、层次聚类 + LLM 归纳

## 安全评估报告

### 总体评分：7.5 / 10

多项主动防御措施到位，但存在 MCP SSE 无速率限制、CORS 默认通配符、X-User-ID 可伪造等结构性问题。

### 关键发现

#### CRITICAL
- CRIT-01: MCP `/sse` 端点无速率限制，可无限连接 DoS
- CRIT-02: MCP CORS 默认 `*` 且无预检验证
- CRIT-03: 主 API CORS 默认通配符

#### HIGH
- HIGH-01: `X-User-ID` Header 可被任意客户端伪造身份（水平越权）
- HIGH-02: 图谱端点缺数据隔离（授权检查全部缺失）
- HIGH-03: `/maintenance/cleanup` 认证关闭时完全开放
- HIGH-04: MCP 服务器缺安全响应头
- HIGH-05: `text/xml` 在 MIME 白名单中，潜在 XXE 载体

#### MEDIUM
- MCP stdio 无身份验证、`X-XSS-Protection` 已废弃、retriever 内部 SystemOwnerID 绕过、Seek 错误被静默丢弃、速率限制仅基于 IP、配置文件 API Keys 明文存储

#### LOW
- List 过滤参数被解析后丢弃、错误消息部分泄露内部信息、无 IdleTimeout、数据目录权限 0755

### 亮点
- 常量时间 API Key 比较、完整安全响应头（主 API）、路径遍历 validatePath 防护、FTS5 注入防护、全参数化 SQL、多层分级速率限制、文件上传双重验证、`__system__` 身份保护

## 架构评估报告

### 总体评分：7.5 / 10

分层清晰、依赖无环、feature-gating 优雅。主要扣分项：MemoryStore 接口臃肿、sqlite.go 超大文件、ParseRouter 硬编码 parser 名称。

### 关键发现

#### CRITICAL
无。

#### HIGH
- H1: `MemoryStore` 接口 26+ 方法违反 ISP
- H2: `sqlite.go` ~1470 行远超 1000 行上限
- H3: `DB() interface{}` 泄露抽象
- H4: `ParseRouter` 硬编码 parser 名称覆盖 ParseResult

#### MEDIUM
- 全局配置单例 `AppConfig`、`LoadConfig()` 默认值 120 行、`search` 依赖 `memory` 仅为一个函数、`ProcessAsync` 裸 goroutine、Processor 混合 CRUD 委托和管线逻辑、`NewManager` 8 个位置参数

#### LOW
- Config 拆分策略不统一、`tokenutil` 包功能单一、Chunk 的 RawContent/Content 语义重叠

### 亮点
- 无循环依赖、Feature Gating nil check、Bootstrap 独立包 main.go 仅 83 行、文档管线 Parser+ParseRouter+Chunker 分层、pkg/ 目录组织得当、LLM 降级链、配置四级优先级、事务一致性保障

### 最高优先级行动项（TOP 3）

1. **[安全] MCP 服务器安全加固** — `/sse` 添加速率限制 + 连接数上限；CORS 默认值改为非通配符；添加安全响应头；`X-User-ID` 改为服务端验证（绑定 API Key）。一次修复覆盖 CRIT-01/02/03 + HIGH-01/04。

2. **[AI/检索] Embedding 输入截断 + 强度加权顺序修复** — 在 embed 层添加 token 上限截断（防 API 400）；将 `ApplyStrengthWeighting` 移到 MMR 之前（防返回结果数不足）。修复 C1 + C2。

3. **[数据库+架构] `memory_tags` 索引 + `MemoryStore` 接口拆分 + `sqlite.go` 拆文件** — V11 迁移添加 `idx_memory_tags_tag_id`；拆分 MemoryStore 为 4 个小接口；`sqlite.go` 按功能域拆为 3-4 个文件。修复 DB-C1 + ARCH-H1/H2。
