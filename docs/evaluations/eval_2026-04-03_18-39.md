## 数据库评估报告

### 总体评分：7.8 / 10

该项目在 SQLite 单文件数据库场景下展现出超出平均水平的工程质量。迁移体系、FTS5 外部内容模式管理、`scanDest` 统一扫描模式均属业界良好实践。主要扣分点集中在两个高优先级索引缺失（`source_ref`、`consolidated_into`）以及 V16 迁移中一个静默失败路径。

### 关键发现

#### CRITICAL

**1. `source_ref` 列无索引，但已成为高频查询路径（B6/B7 方法）**

`ListBySourceRef`、`SoftDeleteBySourceRef`、`RestoreBySourceRef`、`ListTimeline`（带 `source_ref` 过滤）均直接扫描 `memories` 表的 `source_ref` 列，而整个迁移链（V0→V16）以及 `createFreshSchema` 中均未为该列建立任何索引。

在 `source_ref` 为 conversation ID 的场景下，一次会话归档（`SoftDeleteBySourceRef`）会对整张 `memories` 表执行全表扫描，再叠加 FTS5 rowid 逐行清理，线性复杂度在数据量增长后会急剧恶化。

修复：
```sql
CREATE INDEX idx_memories_source_ref ON memories(source_ref) WHERE source_ref != '' AND deleted_at IS NULL;
```

**2. `consolidated_into` 列无索引，`ListConsolidatedInto` 全表扫描**

`ListConsolidatedInto` 是归纳回溯的核心查询，过滤条件 `WHERE consolidated_into = ?` 无任何索引支持。

修复：
```sql
CREATE INDEX idx_memories_consolidated_into ON memories(consolidated_into) WHERE consolidated_into != '' AND deleted_at IS NULL;
```

#### HIGH

**3. V15→V16 迁移：`ALTER TABLE DROP COLUMN` 静默失败，旧列可能残留**

`migrateV15ToV16` 在 `derived_from` 列删除失败时只记录 Warn 日志，不中断迁移，允许旧 JSON 列继续存在。建议替换为完整的 table recreate 方式。

**4. `PurgeDeleted` 遗漏 `memory_derivations` 的手动级联清理**

`PurgeDeleted` 手动删除 `memory_tags` 和 `memory_entities`，但未包括 `memory_derivations`。虽然有 FK CASCADE，但 `PRAGMA foreign_keys=ON` 是连接级设置，连接池回收重建时可能丢失。建议 PRAGMA 改为 DSN 参数写法，同时显式清理 `memory_derivations`。

**5. `ListEntities` 无 limit 上界**

仅有下界检查，缺少 `if limit > 200 { limit = 200 }` 上界。

**6. `visibilityCondition` 在 `TeamID=""` 时行为宽松**

空 identity 可以读到所有团队的数据，违反最小权限原则。

#### MEDIUM

**7. `RestoreBySourceRef` 中 FTS5 重建 `defer restoredRows.Close()` 与事务游标冲突**

应在 for 循环前显式 `Close()`，移除 `defer`。

**8. `sanitizeFTS5Query` 的 bigram 扩展在长查询时二次方膨胀**

建议对 bigram 扩展设置词数上限（如 `len(filtered) <= 5`）。

**9. `ListTimeline` 缺少 OFFSET/cursor 分页支持**

#### LOW

**10. `async_tasks` 表未使用** — V8 创建了表但无业务代码读写。

### 亮点

- **迁移体系成熟度高**：16 个版本增量迁移全部事务内执行，`createFreshSchema` 快路径避免历史回放
- **`scanDest` 三件套**：35 列统一封装，新增列只改一处
- **FTS5 external content 模式管理正确**：所有写入同事务内 FTS5 同步
- **WAL + 连接池参数合理**：`MaxOpen=5`、`mmap_size=256MB`、`busy_timeout=5000`

## AI/检索评估报告

### 总体评分：8.6 / 10

架构成熟、工程质量上乘的企业级记忆检索系统。三路检索 + 加权 RRF 融合的设计在开源同类项目中属于领先水平，B7 新增功能完成度高，记忆演化模型设计前瞻。

### 关键发现

#### CRITICAL

无。核心检索路径和数据一致性逻辑无严重缺陷。

#### HIGH

**1. Consolidation 聚类算法复杂度为 O(n^3)**

`agglomerativeClustering` 使用 naive 贪心合并，`maxMemoriesPerRun=100` 时有性能风险。建议预计算距离矩阵 + 最小堆，或 n>50 时切换近似方案。

**2. Embedding 缓存 LRU `touchLocked` 为 O(n) 线性扫描**

应改用 `container/list` + map 实现 O(1) LRU。

**3. Reflect Engine 缺少并发控制和全局 token 消耗限制**

无 semaphore 或速率限制，多用户同时触发可打满 LLM provider。

#### MEDIUM

**4. 向量去重搜索绕过可见性过滤** — 用户 A 的新记忆可能因与用户 B 私有记忆相似而被判重复。

**5. HyDE 通道使用 `context.Background()` 脱离请求生命周期** — 主请求取消后 LLM 调用不会取消。

**6. SessionSummarizer 硬编码 200 条上限** — 大型会话可能截断。

**7. Consolidation 直接 mutate 原始 Memory 对象** — 违反 immutability 规范。

#### LOW

**8. RRF 融合后 Source 标记丢失**（统一标记为 "hybrid"）

**9. LineageTracer consolidated 方向语义混淆** — consolidated sources 放在 Children 而非 Sources 中。

**10. ExperienceRecaller 误用变量名** — `noRetry` 变量传给 `RerankEnabled`。

### 亮点

- **五通道加权 RRF 架构**：FTS5 + Qdrant + Graph + HyDE + Temporal，教科书级混合检索
- **意图感知动态权重调整**：规则 + LLM 双路分类，CJK 适配
- **Reflect 的 adaptive top-K + prior round summary**：真正的渐进式推理
- **三级 fallback 解析贯穿全系统**：JSON -> regex -> LLM retry -> raw
- **完整的记忆生命周期模型**：五级 retention tier + 指数衰减 + 自动晶化

## 安全评估报告

### 总体评分：7.8 / 10

代码质量整体较高，具备多项主动防御措施。评分主要被 B6/B7 批量操作缺少所有者验证、LLM 内容直接流入日志、以及 MCP 速率限制粒度不足拉低。

### 关键发现

#### CRITICAL

**1. by-source 批量写操作无 owner 验证（越权操作）**

`SoftDeleteBySourceRef` 和 `RestoreBySourceRef` 的 Store 层 SQL 只按 `source_ref` 匹配，没有 `team_id`/`owner_id` 过滤。攻击者可批量软删除或恢复**其他租户的所有记忆**。

**2. Summarize 端点无跨 team 归属校验**

`SummarizeBySourceRef` 可触发 LLM 对他人记忆做总结，构成跨 team 信息采集。

#### HIGH

**3. ListBySourceRef limit 无上界** — 可传 `limit=10000000`（Store 层有 200 防线但 API 层不一致）。

**4. raw_content 直接写入日志** — Extractor 和 Reflect 的 LLM 输出含用户 PII。

**5. MCP SSE 速率限制全局粒度** — 无 per-session 限制，可被单用户独占。

**6. MCP CORS 无动态 origin 校验** — 若配为 `*` 则完全开放。

#### MEDIUM

**7. Reflect 自动保存时 OwnerID 为空** — mental_model 记忆对 team 内全部成员可读。

**8. Summarizer 传给 LLM 的 context name 无 prompt 注入防护**。

**9. Summarize 端点未应用 llmRateLimit**。

#### LOW

**10. MCP 未配置 API token 时仅 Warn 不阻止启动**。

### 亮点

- **全路径参数化查询** + FTS5 查询净化
- **常量时间 API Key 比较**（`crypto/subtle.ConstantTimeCompare`）
- **分级速率限制**：写 20rps、搜索 10rps、LLM 2rps
- **完整 Security Headers** + 文件上传路径遍历防护 + Magic bytes MIME 检测
- **MCP 握手强制** + `__system__` 身份保护

## 架构评估报告

### 总体评分：7.5 / 10

分层清晰、依赖单向、接口设计合理。但存在 B7 组件未完成布线、部分质量关卡规则未落地、以及 RouterDeps 膨胀等值得关注的问题。

### 关键发现

#### CRITICAL

**1. B7 组件未布线 — SessionSummarizer/LineageTracer/ExperienceRecaller 永远为 nil**

`RouterDeps` 已声明字段，handler 已编写完整逻辑，但 `bootstrap/wiring.go` 和 `cmd/server/main.go` 均未实例化这三个组件。所有 B7 端点永远返回错误或静默失效。

**2. SoftDeleteBySourceRef/RestoreBySourceRef 缺少归属校验**（同安全 C1）

#### HIGH

**3. search 包对 memory 包的直接依赖**

`ApplyStrengthWeighting` 是纯函数，应迁移到 `pkg/` 消除 `search -> memory` 依赖。

**4. RouterDeps 膨胀（15+ 字段）** — God Config 反模式。

**5. ListBySourceRef 缺少 limit 上界**（违反质量关卡 #5）。

#### MEDIUM

**6. Summarizer 硬编码 200 条上限** — 应可配置化或按 token budget 动态计算。

**7. Summarize/SummarizeBySourceRef 约 80% 代码重复** — 应提取公共方法。

**8. SessionSummarizer 持有 Manager 引用** — 应注入 `MemoryCreator` 接口替代。

**9. bootstrap/wiring.go 作为单一布线文件过度集中**。

#### LOW

**10. Nil-check 门控模式不一致** — 部分路由级（404）、部分 handler 级（400）。

### 亮点

- **依赖流向严格单向**：`cmd -> api -> reflect/memory/search -> store -> model`
- **接口组合设计优秀**：`MemoryStore` = 5 个子接口，消费者按需依赖
- **Nil-check 特性门控**务实有效
- **Store 层规范化程度高**：14 个文件按职责清晰拆分
- **可插拔架构成熟**：tokenizer/LLM/检索/文档解析均实现策略模式

## 综合总结

| # | 问题 | 影响维度 | 修复复杂度 |
|---|------|----------|-----------|
| 1 | **B7 组件未布线** — Summarizer/LineageTracer/ExperienceRecaller 永远 nil | 架构 | LOW（3 行构造 + 赋值） |

### 最高优先级行动项

1. **补全 B7 布线**：在 `bootstrap/wiring.go` 中实例化 `SessionSummarizer`、`LineageTracer`、`ExperienceRecaller`，赋值到 `RouterDeps`。预计 30 行代码。

2. **修复批量操作授权**：`SoftDeleteBySourceRef` / `RestoreBySourceRef` 的 Store 层 SQL 加入 `team_id` + `owner_id` 过滤，Manager 层方法签名增加 `identity` 参数。同步修复 `SummarizeBySourceRef` 的跨 team 校验。

3. **补缺索引 + LLM 日志脱敏**：出 V17 迁移加 `source_ref` 和 `consolidated_into` 索引；Extractor/Reflect 的 `raw_content` 日志截断至 100 字符。
