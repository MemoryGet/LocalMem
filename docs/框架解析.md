# IClude 框架层次解析

> 更新日期：2026-04-14  
> 基于当前代码库实际状态，非设计文档推测。

---

## 整体架构层次

```
cmd/                          ← 入口层（3个二进制）
  server/                     → HTTP API 服务 (port 8080)
  mcp/                        → MCP 协议服务 (port 8081 / stdio)
  cli/                        → Claude Code 集成 hooks

internal/bootstrap/           ← 装配层（核心）
  wiring.go                   → InitDeps() 返回 Deps 结构体，聚合所有组件

internal/api/                 ← HTTP 层
  router.go + *_handler.go    → Gin 路由 + 17 个 handler（50 个路由）

internal/mcp/                 ← MCP 层
  tools/ resources/ prompts/  → 11 个 MCP 工具

internal/memory/              ← 业务核心（3883行，20文件）
internal/search/              ← 检索层（6232行）
internal/reflect/             ← 推理层
internal/runtime/             ← 会话运行时
internal/queue/               ← 异步任务队列
internal/store/               ← 存储层（14个接口，6821行SQLite实现）
internal/heartbeat/           ← 自治巡检
internal/scheduler/           ← 定时调度
internal/document/            ← 文档摄取
internal/llm/ embed/          ← AI 基础设施
internal/config/ logger/      ← 基础设施（全局单例）
```

**依赖流向（严格单向）：**

```
cmd/*
  └─ bootstrap.Init()
       └─ api / mcp
            └─ reflect, memory, search, document, runtime
                 └─ store（接口）
                      └─ model, pkg/*
```

## 入口层 `cmd/`

| 二进制 | 做什么 |
|--------|--------|
| `cmd/server/` | 启动 HTTP API 服务（port 8080），调用 `bootstrap.Init()` 拿 Deps，传给 `api.SetupRouter()` |
| `cmd/mcp/` | 启动 MCP 服务，支持两种传输：`--stdio`（Claude Code/Codex 用）或 SSE HTTP（port 8081） |
| `cmd/cli/` | 三个 Claude Code hook 二进制：`hook_session_start`、`hook_session_stop`、`hook_capture`，在 AI 工具生命周期节点触发 API 调用 |

## 装配层 `internal/bootstrap/`

**唯一职责：把所有组件按正确顺序初始化，组装成 `Deps` 结构体。**

`Init()` 内部顺序：

1. Embedder（Qdrant 启用时）
2. Stores（SQLite + Qdrant）
3. LLM Provider
4. 业务 Manager 群（Graph/Context/Extractor/Consolidator）
5. 检索层（Retriever + CascadeRetriever + ExperienceRecaller）
6. 上层服务（Reflect/Doc/Summarizer/Lineage/Runtime）
7. 异步队列 Queue
8. 调度器 Scheduler

`cmd/server` 和 `cmd/mcp` 都调用它，共享同一套初始化逻辑。

`Deps` 结构体字段（nil-check 门控可选组件）：

```go
Stores, MemManager, Retriever,
ContextManager, GraphManager, DocProcessor, DocFileStore,
ReflectEngine, TagManager, Extractor, Summarizer,
LineageTracer, ExperienceRecaller,
SessionService, FinalizeService, RepairService,
Scheduler, SchedCancel, Queue, Config
```

## HTTP 层 `internal/api/`

**职责：参数解析 → 身份注入 → 调用业务层 → 统一响应格式。**

50 个路由，分 10 组 handler：

| Handler | 路由组 | 做什么 |
|---------|--------|--------|
| `memory_handler` | `/v1/memories` | CRUD + 软删除/恢复/强化 |
| `session_handler` | `/v1/memories/by-source/`、`/sessions/*/summarize`、`/:id/lineage` 等 | 会话关联操作、溯源链 |
| `batch_handler` | `/v1/memories/batch` | 批量按 ID 获取 |
| `search_handler` | `/v1/retrieve`、`/v1/timeline` | 触发三路 RRF 检索 |
| `conversation_handler` | `/v1/conversations` | 对话批量入库 |
| `context_handler` | `/v1/contexts` | 层级上下文树（物化路径） |
| `tag_handler` | `/v1/tags`、`/v1/memories/:id/tags` | 标签 CRUD + 关联 |
| `graph_handler` | `/v1/entities`、`/v1/entity-relations`、`/v1/memory-entities` | 知识图谱 |
| `document_handler` | `/v1/documents` | 文档上传/列表 |
| `reflect_handler` + `extract_handler` | `/v1/reflect`、`/v1/memories/:id/extract` | LLM 推理 + 实体抽取 |
| `scope_policy_handler` | `/v1/scope-policies` | Scope 权限策略 |

中间件栈：Auth → IdentityMiddleware → RateLimit（写 20rps / LLM 2rps）→ CORS → MaxBodySize(4MB)

## MCP 层 `internal/mcp/`

**职责：把 MCP 协议调用翻译为业务层调用，管理 Session 生命周期。**

11 个工具：

| 工具 | 做什么 |
|------|--------|
| `iclude_retain` | 存记忆（调 Manager.Create） |
| `iclude_recall` | 检索记忆（调 Retriever） |
| `iclude_scan` | 列出记忆摘要（渐进式披露） |
| `iclude_fetch` | 获取单条记忆全文 |
| `iclude_reflect` | 触发 Reflect Engine 多轮推理 |
| `iclude_timeline` | 时间线查询 |
| `iclude_ingest_conversation` | 对话批量入库 |
| `iclude_create_session` | 创建会话（写 sessions 表） |
| `iclude_finalize_session` | 终态处理（触发摘要生成） |
| + 2 个其他工具 | — |

Session 状态机（7 状态）：`created → active → finalizing → finalized / abandoned / error / unknown`

## 业务核心层 `internal/memory/`

**职责：记忆的完整生命周期管理，是写路径的唯一入口。**（3883 行，20 个文件）

| 组件 | 文件 | 做什么 |
|------|------|--------|
| `Manager` | `manager.go` (226行) | CRUD 调度中心。Create 流程：NoiseFilter → Dedup → SQLite写 → Qdrant写 → 生成摘要 → 入队实体抽取 |
| `Extractor` | `extractor.go` (665行) | LLM 驱动实体/关系抽取。3级解析兜底：JSON → 正则 → LLM重试 → raw fallback |
| `EntityResolver` | `entity_resolver.go` (267行) | 向量驱动实体解析：分词 → 向量质心 → 近邻匹配（替代 LLM 抽取，无 API 消耗） |
| `CentroidManager` | `centroid_manager.go` | 维护各实体的向量质心（多向量均值），供 EntityResolver 使用 |
| `GraphManager` | `graph_manager.go` (248行) | 知识图谱 CRUD：实体/关系/记忆-实体关联 |
| `ContextManager` | `context_manager.go` (142行) | 层级上下文树（物化路径），存行为字段（mission/directives/disposition） |
| `Consolidator` | `consolidation.go` (419行) | 记忆归纳：找相似记忆 → LLM合并 → 写 memory_derivations 溯源 |
| `SessionSummarizer` | `summarizer.go` (250行) | LLM 生成会话摘要，写入 `mental_model` 类记忆 |
| `LineageTracer` | `lineage.go` (123行) | 溯源链查询（derived_from / consolidated_into / 完整 lineage 树） |
| `TagManager` | `tag_manager.go` | 标签 CRUD + 记忆-标签关联 |
| `NoiseFilter` | `noise_filter.go` | 入库前预过滤噪声（过短、无意义内容） |
| `Promoter` | `promoter.go` (161行) | 候选记忆晋升（candidate_for 字段驱动） |
| `AccessTracker` | `access_tracker.go` | 异步批量更新访问计数（防写热点） |
| `Dedup` | `dedup.go` (119行) | 内容哈希去重（依赖 UNIQUE 约束，TOCTOU 安全） |

## 检索层 `internal/search/`

**职责：多路融合检索，从查询到结果的全链路。**（6232 行）

```
查询
  └─ 预处理（preprocess.go，432行）
       ├─ 分词 → 同义词扩展 → HyDE（权重可配，默认0.8）
       └─ LLM 查询增强 → 意图分类
  └─ IntentClassifier（intent.go）→ 4 类意图（entity/temporal/conceptual/default）
  └─ CascadeRetriever（cascade.go，223行）→ 按意图选 Stage 序列
  └─ Retriever（retriever.go，614行）→ 三路并行
       ├─ FTSStage → SQLite FTS5 BM25
       ├─ VectorStage → Qdrant 向量
       └─ GraphStage → 实体关联 + LLM fallback
  └─ MergeWeightedRRF（k=60）→ 融合排序
  └─ 后处理 Stage 链（stage/）
       ├─ weight.go（167行）→ 强度加权 + kind/class 权重（weightCap=2.0）
       ├─ mmr.go（176行）→ MMR 多样性重排
       ├─ rerank_llm.go（292行）→ LLM 重排
       ├─ rerank_graph.go（263行）→ 图关联重排
       ├─ rerank_overlap.go（185行）→ 重叠去重重排
       ├─ rerank_remote.go（290行）→ 远程重排服务
       ├─ disclosure.go（156行）→ 渐进式披露（token预算自适应）
       ├─ temporal.go（152行）→ 时间独立 RRF 通道
       └─ trim.go → token预算截断
```

`ExperienceRecaller`：创建记忆时主动召回相关 `procedural` 记忆，作为 hint 返回给调用方（B7 特性）。

## 推理层 `internal/reflect/`

**职责：对已有记忆做多轮 LLM 推理，产出 mental_model 结论。**

- 接收查询 → 检索相关记忆 → 多轮 LLM 对话（最多 `max_rounds`）
- 查询去重防死循环
- 结论可 auto_save 为 `mental_model` 类记忆
- 同 Extractor 一样的 3 级解析兜底

---

## 会话运行时层 `internal/runtime/`

**职责：管理 AI 工具会话的完整生命周期，保证幂等和容错。**

| 组件 | 做什么 |
|------|--------|
| `SessionService` | 7 状态状态机（created/active/finalizing/finalized/abandoned/error/unknown），CRUD + 状态转换 |
| `FinalizeService` | 幂等终态处理：ingest 对话 → 生成摘要记忆。依赖 `session_finalize_state` 表做 at-least-once 保证 |
| `RepairService` | 扫描卡死会话 → 追踪 repair attempts → 超限转 abandoned。每 10 分钟由 Scheduler 触发 |
| `ExecLauncher` | 启动 4 种宿主进程（claude_code/codex/opencode/generic） |

---

## 异步队列层 `internal/queue/`

**职责：SQLite 持久化任务队列，解耦写路径与耗时操作。**

- 状态：`pending → processing → completed / failed`
- 支持 retry（max_retries 可配）和 scheduled_at 延迟执行
- `Manager.Create()` 通过 `TaskEnqueuer` 接口入队实体抽取任务（不直接依赖 queue 包）
- Worker 轮询 poll_interval，stale_timeout 超时重置为 pending

## 存储层 `internal/store/`

**职责：数据持久化，全部通过接口隔离，业务层不感知实现。**（6821 行 SQLite 实现）

**14 个接口：**

| 接口 | 后端 | 做什么 |
|------|------|--------|
| `MemoryStore`（组合4子接口） | SQLite | 记忆 CRUD + FTS5搜索 + 生命周期 |
| `DerivationStore` | SQLite | 记忆溯源 junction 表 |
| `VectorStore` | Qdrant | 向量存取 |
| `Embedder` | OpenAI/Ollama | 文本→向量 |
| `ContextStore` | SQLite | 层级上下文树 |
| `TagStore` | SQLite | 标签 + 记忆关联 |
| `GraphStore` | SQLite | 实体/关系/记忆-实体 |
| `CandidateStore` | SQLite (V27+) | 候选实体 + 质心向量 |
| `DocumentStore` | SQLite | 文档元数据 + chunk |
| `SessionStore` | SQLite (V18+) | 会话记录 |
| `SessionFinalizeStore` | SQLite (V19+) | 终态幂等状态 |
| `TranscriptCursorStore` | SQLite (V20+) | 对话文件读取游标 |
| `IdempotencyStore` | SQLite (V21+) | 幂等键（防重复请求） |
| `ScopePolicyStore` | SQLite (V24+) | Scope 权限策略 |

迁移：V0→V27，两条路径（新库用 `createFreshSchema()`，老库走增量迁移）。

## 自治层 `internal/heartbeat/` + `internal/scheduler/`

**职责：无需人工干预的后台维护任务。**

| 任务 | 触发方式 | 做什么 |
|------|---------|--------|
| `decay audit` | Heartbeat 定时 | 按衰减率降低记忆 Strength |
| `orphan cleanup` | Heartbeat 定时 | 清理孤儿实体/关系 |
| `contradiction detection` | Heartbeat 定时（可配）| LLM 检测矛盾记忆 |
| `excerpt backfill` | Heartbeat 定时 | 补全缺失摘要 |
| `session-repair` | Scheduler 每 10 分钟 | 调 RepairService |
| `access flush` | Scheduler 定时 | 批量写入访问计数 |
| `consolidation` | Scheduler 定时 | 触发记忆归纳 |

两者均 config-gate（`heartbeat.enabled`/`scheduler.enabled`，默认 false）。

## 文档摄取层 `internal/document/`

**职责：文件→记忆的完整异步管线。**

```
上传（FileStore.Save）
  → 解析（Docling → Tika fallback）
  → 分块（MarkdownChunker 或 TextChunker，512 token / 50 overlap）
  → 向量嵌入
  → 写入 MemoryStore（经 Manager 层，保证 embedding/副作用）

状态流：pending → parsing → chunking → embedding → ready / failed
并发：goroutine + semaphore（默认 3 并发）
Config-gate：document.enabled: true + docling/tika Docker sidecar
```

## 数据流示例：一次 `iclude_retain`

```
MCP tool → Manager.Create()
  ├─ NoiseFilter     过滤噪声
  ├─ Dedup           内容哈希去重（UNIQUE 约束保障）
  ├─ SQLite 写入     主存储
  ├─ Qdrant 写入     副存储（失败不回滚 SQLite）
  ├─ generateExcerpt 同步生成摘要
  └─ Queue.Enqueue   异步实体抽取
                       └─ Worker 消费 → Extractor.Extract()
                                          └─ LLM → 实体/关系 → GraphStore
```
